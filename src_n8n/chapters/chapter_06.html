Chapter 6: Expressions & the n8n Data Model

The n8n Item Model

To write expressions, we first have to understand what an "Item" actually is in memory. When a node outputs data-say, fetching a row from Postgres-n8n wraps that data in a standard Item object containing two root keys: json and binary. 

// What a single Item looks like under the hood:
{
  "json": {
    "user_id": 1234,
    "email": "dev@company.com",
    "name": "Jane Developer"
  },
  "binary": {
    "profile_pic": {
      "mimeType": "image/png",
      "fileType": "png",
      "fileName": "avatar.png",
      "data": "...base64_string..."
    }
  }
}

Everything we do in n8n visually is just manipulating the json or binary object of these Items.

Expression Syntax

Expressions dynamically evaluate JavaScript inside almost any text field in a node. We surround expressions with double curly braces: {{ }}.

  • {{ $json.email }} - Grabs the email from the current item.
  • {{ $json.user_id + 100 }} - We can perform pure math.
  • {{ $json.name.toLowerCase() }} - We can use strict native JS methods.
  • {{ $json.address?.city }} - Crucial: Use optional chaining (?.) to prevent the entire workflow from crashing if address is null on one of the items.
Common Mistake: Do not write {{ $json['first-name'] }} if we can avoid it. n8n's expression editor autocomplete heavily relies on dot notation. Only use bracket notation for keys with spaces or hyphens.

Accessing Previous Node Output

Often, we are five nodes deep into a workflow and we need the ID from the very first trigger node. We don't have to pass it manually through every step. We can query any previous node's output directly.

// Standard cross-node reference:
{{ $('Stripe Trigger').item.json.data.object.id }}

// Accessing the exact first item of a previous node (bypasses current item iteration mapping):
{{ $('Get CRM Users').first().json.email }}

// Accessing ALL items from a previous node as an array:
{{ $('Database Query').all().map(item => item.json.id) }}

Built-in Variables

n8n exposes several global variables inside expressions to give we contextual awareness of the execution environment.

Variable What it does
$json / $binary The data for the current iteration/item passing through the node.
$input The complete array of all items entering the node ($input.all()).
$env("KEY") Pulls a value from our .env file (e.g., {{ $env("N8N_HOST") }}).
$workflow.id The unique UUID of the current workflow (great for creating automatic error Slack links).
$execution.id The ID of this specific run.
$now / $today Luxon DateTime objects bound to the server's timezone.
Security Tip: n8n strictly blocks $env("N8N_ENCRYPTION_KEY") and other sensitive internal variables from being exposed in expressions to prevent malicious blueprint imports from stealing our keys.

JMESPath and Data Searching

When dealing with massive, deeply nested JSON responses (like from the AWS or Jira APIs), mapping arrays manually using JS .filter() or .map() inside an expression gets messy.

n8n natively supports JMESPath using the $jmespath() function to query and slice JSON elegantly.

// Given an array of 500 users attached to $json.users, extract an array of ONLY the names
// of users who are strictly active:

{{ $jmespath($json, "users[?status=='active'].name") }}

Luxon: Time Manipulation

Writing new Date().toISOString() works, but manipulating time zones, adding days, and formatting strings in vanilla JS is famously brutal. n8n bundles the Luxon library globally.

Whenever we use $now or $today, we are working with a Luxon DateTime object.

// Get the exact time exactly 7 days from right now in UTC format
{{ $now.plus({ days: 7 }).toUTC().toISO() }}

// Get the last day of the current month formatted nicely for an email
{{ $today.endOf('month').toFormat('MMMM dd, yyyy') }}

// Parse an ugly API timestamp string and convert it to our timezone
{{ $now.fromISO($json.created_at).setZone('America/New_York').toFormat('HH:mm') }}

Handling Null / Missing Fields Safely

If Node A fetches 10 users, and User #4 doesn't have a company block, an expression like {{ $json.company.name }} will crash the entire node with a TypeError: Cannot read properties of undefined.

Always use modern JS fallback patterns in expressions when dealing with optional API data:

// 1. Optional Chaining (Prevents crash, returns undefined)
{{ $json.company?.name }}

// 2. Nullish Coalescing (Provides a fallback instead of undefined)
{{ $json.company?.name ?? 'Independent Contractor' }}

// 3. Logical OR (Falls back on empty strings '' as well as null/undefined)
{{ $json.company?.name || 'Unknown' }}
