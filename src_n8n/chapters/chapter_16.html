Chapter 16: Building Custom Nodes

When To Build a Custom Node

We've already seen that n8n offers multiple ways to integrate with external systems. So when should we actually write a custom TypeScript node from scratch instead of just using an HTTP Request or a Sub-Workflow?

Approach Best For Setup Time
HTTP Request Node One-off API calls, internal endpoints. 2 minutes
Sub-Workflow Reusable logic combining multiple API calls. 10 minutes
Custom Node Providing a beautiful UI for non-technical users to access our proprietary internal API. ~1-2 days
💡 Tip: If we are the only developers using the n8n canvas, stick to the HTTP Request node. We only build custom nodes when we want to abstract away API complexity (like handling pagination or tricky OAuth handshakes) so our marketing or operations team can drag-and-drop our internal API just like they do with Stripe.

Project Structure & Dev Environment

n8n provides a starter template exactly for this purpose. We don't start from a blank folder. Instead, we clone their starter repo which comes pre-configured with TypeScript, ESLint, and the n8n compiler scripts.

A typical node repository (often named n8n-nodes-mycompany) looks like this:

n8n-nodes-mycompany/
├── package.json
├── credentials/
│   └── MyApiApi.credentials.ts    # Defines how users authenticate
└── nodes/
    └── MyApi/
        ├── MyApi.node.ts          # The core logic and UI definitions
        └── GenericFunctions.ts    # Helper functions for making HTTP calls

During development, we need a way to run n8n locally while it watches our TypeScript files for changes. We achieve this by linking our local repo using npm link.

⚠️ Warning: Avoid writing nodes in plain JavaScript. n8n relies heavily on TypeScript interfaces (like INodeType) to extract UI definitions during the build step. Without TypeScript, the build process will fail.

The INodeType Interface Anatomy

Every node we build must implement the INodeType interface. This class does two things: it defines the UI properties (what the node looks like, what inputs it requires), and it defines the Execute function (what the node actually does).

The description object dictates the frontend UI:

// nodes/MyApi/MyApi.node.ts
import { INodeType, INodeTypeDescription } from 'n8n-workflow';

export class MyApi implements INodeType {
    description: INodeTypeDescription = {
        displayName: 'My Internal API',
        name: 'myApi',
        icon: 'file:myApi.svg', // Local SVG file for the node icon
        group: ['transform'],
        version: 1,
        description: 'Consume our internal microservices',
        defaults: {
            name: 'My Internal API',
        },
        inputs: ['main'],
        outputs: ['main'],
        credentials: [
            {
                name: 'myApiApi',
                required: true,
            },
        ],
        // The properties array defines the actual form fields in the UI
        properties: [
            {
                displayName: 'Operation',
                name: 'operation',
                type: 'options',
                noDataExpression: true,
                options: [
                    {
                        name: 'Get User',
                        value: 'getUser',
                    },
                    {
                        name: 'Create User',
                        value: 'createUser',
                    },
                ],
                default: 'getUser',
            },
            {
                displayName: 'User ID',
                name: 'userId',
                type: 'string',
                required: true,
                displayOptions: {
                    show: {
                        operation: ['getUser'], // Only shows if operation == 'getUser'
                    },
                },
                default: '',
            }
        ],
    };
    
    // The execute function runs when the node actually fires
    async execute(this: IExecuteFunctions): Promise<INodeExecutionData[][]> {
        // Implementation logic here...
    }
}

Returning Items Correctly

Inside the execute method, we loop over the incoming items, process our API calls, and return the mutated data. We must return an array containing an array of INodeExecutionData objects. This matches n8n's internal data model.

Here is a real example of the execution logic for our internal API node:

async execute(this: IExecuteFunctions): Promise<INodeExecutionData[][]> {
    const items = this.getInputData();
    const returnData: INodeExecutionData[] = [];

    // Loop over every item entering the node
    for (let i = 0; i < items.length; i++) {
        try {
            const operation = this.getNodeParameter('operation', i) as string;
            
            if (operation === 'getUser') {
                const userId = this.getNodeParameter('userId', i) as string;
                
                // Fetch our credentials to grab the API key
                const credentials = await this.getCredentials('myApiApi');
                
                // We use n8n's built-in this.helpers.request to handle the HTTP call
                const response = await this.helpers.request({
                    method: 'GET',
                    url: `https://api.mycompany.com/users/${userId}`,
                    headers: {
                        'Authorization': `Bearer ${credentials.apiKey}`
                    },
                    json: true,
                });

                // Wrap the API response in n8n's expected JSON structure
                returnData.push({ json: response });
            }
        } catch (error) {
            // Handle error logic: If "Continue on Fail" is set, don't crash
            if (this.continueOnFail()) {
                returnData.push({ json: { error: error.message } });
                continue;
            }
            throw error;
        }
    }

    return [returnData];
}
Important: Notice how we loop through items.length. A node might receive 1 item from a Webhook, or 500 items from a Postgres trigger. We must process each item individually in the execute loop to ensure data consistency.

Publishing & Community Nodes

Once we have built and tested our node locally, we can publish it as standard Node packages to npm. Community nodes are simply npm packages prefixed with n8n-nodes-.

To install a community node in our production n8n instance (like Docker), we can go to Settings → Community Nodes in the n8n UI, input the npm package name, and n8n will automatically download and install it into our running instance.

⚠️ Warning: When self-hosting via Docker Compose, installing via the UI works, but destroys immutability. If the container crashes and rebuilds, the node is gone. Make sure to define community nodes directly in our docker-compose.yml or bake them into a custom Dockerfile via RUN npm install -g n8n-nodes-mycompany.
