The Canvas Overview

When we open n8n, we are greeted by an infinite grid dotted with nodes. The canvas is where we visually program our automation. Think of it as a spatial IDE where functions (Nodes) are visually linked by data flow (Connections).

  • Nodes: The building blocks. A node represents a single operation-like receiving a webhook, fetching a Postgres row, or running a snippet of JavaScript.
  • Connections (The Wires): The lines connecting the right side (output) of one node to the left side (input) of another. They dictate the execution path.
  • Execution Paths: n8n runs nodes sequentially from left to right. If a node outputs multiple "items" (like 5 rows from a database), the next node runs 5 times automatically-once for each item.

Adding and Connecting Nodes

To add a node, click the large + button or right-click anywhere on the canvas. A search panel will appear allowing us to search for integrations (like Slack, GitHub, HTTP Request) or logical operations (like Switch, IF, Loop).

To connect them, click and drag the small grey dot on the right side of our source node to the left side of our destination node. To delete a connection, hover over the wire and click the trash can icon.

Developer Tip: We can branch executions easily. If we drag the output of Node A to both Node B and Node C, n8n will run Node B and Node C in parallel when Node A finishes.

The Node Panel Deep Dive

Double-clicking any node opens its configuration panel on the right. This panel is split into three crucial tabs:

  1. Parameters: The core logic. This is where we configure the API endpoint, specify the database query, or write the JavaScript code. For app nodes, this is where we map fields securely using expressions.
  2. Settings: Metadata for the execution. Here we can configure what happens if the node fails (Continue On Fail), if the node should execute once or per-item, and setup automatic retry logic (e.g., Retry on Fail: 3 times).
  3. Notes: Plain text documentation. Always leave notes explaining why a node exists, perfectly matching the concept of code comments.

The Expression Editor: {{ }} Syntax

We cannot hardcode everything. When we want Node B to dynamically use data generated by Node A (e.g., Node A fetches a user ID, Node B updates that user in a database), we use Expressions.

n8n's expression syntax uses double curly braces: {{ }}. Anything inside the braces evaluates as pure JavaScript.

// Accessing the output of the current item
{{ $json.email }}

// String interpolation using JS template literals
{{ `Hello ${$json.firstName}, your ID is ${$json.id}` }}

// Accessing data from a specific previous node
{{ $('Get User Data').item.json.email }}

// Running native JS methods
{{ $json.timestamp.split('T')[0] }}
Common Mistake: n8n evaluates the expression per Item. If the previous node outputs 10 items, the expression {{ $json.email }} will run 10 times, dynamically resolving to a different email each time. Don't try to loop over $json manually using .map() unless we specifically need to process the entire batch at once.

Pinning Data for Testing

A massive pain point in automation is building workflows that rely on external webhooks. If we are building a Stripe integration, we don't want to make an actual purchase every time we test a node.

n8n solves this with Pinned Data. We can trigger the webhook once, click the "Pin" icon on the node's output, and n8n will freeze that JSON payload. Every time we click "Test Workflow," n8n will pretend the webhook just fired with the pinned data, letting us build the rest of the workflow offline and instantaneously.

Workflow Settings

By clicking the gear icon in the toolbar, we access the global Workflow Settings. These variables override system defaults for this specific automation:

  • Error Workflow: We can specify a completely separate "Error Handler" workflow. If any node in the current workflow fails, n8n halts execution and passes the error trace to the Error Workflow (which could send a Slack alert or PageDuty ping).
  • Timezone: Overrides the GENERIC_TIMEZONE env variable. Crucial if a specific workflow handles localized cron jobs.
  • Execution Timeout: By default, workflows run until they finish. If we have an HTTP node hitting a frozen server, our workflow might hang indefinitely. Set a max timeout (e.g., 5 minutes) to forcefully kill hanging logic.

Execution History

The "Executions" tab on the left sidebar is our dashboard for observability. Every time an active workflow runs, a record is dropped here.

  • Success (Green): The workflow completed normally. We can click into it to see the exact JSON payload traveling between every single node at every step.
  • Error (Red): The workflow crashed. n8n highlights the exact node that threw the error. We can see the HTTP status code, the stack trace, and the exact data the node was processing when it died.
In Plain English: The Execution History is an automatically generated, visual stack trace with full memory dumps at every step. It turns a 2-hour debugging session into a 30-second fix.

The Credential Manager

Never hardcode API keys in HTTP Request nodes. n8n houses a secure Credential Manager (accessed via the left sidebar). We create a credential once (e.g., "Production Stripe Key"), and n8n encrypts it in the database using the N8N_ENCRYPTION_KEY we defined in Chapter 2.

To use it, drop an HTTP node on the canvas, set Authentication to "Predefined Credential Type," and select our saved key. The API key is securely injected at runtime.
