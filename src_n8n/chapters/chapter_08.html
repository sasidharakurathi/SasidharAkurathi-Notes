The Philosophy of Credentials in n8n

Hardcoding an API key into an HTTP node or a Code node is strongly anti-pattern in n8n. If that workflow is exported, or if we rotate the key, we have to hunt down every single node where we pasted that string.

n8n isolates authentication into a global Credential Manager. We create a credential once, n8n encrypts it in the database, and we simply "reference" it inside our nodes.

Credential Types

n8n splits credentials into two broad categories:

  • App-Specific Credentials: Built specifically for official integration nodes (e.g., "Stripe API Key", "Slack OAuth2"). These handle the exact headers and formatting the target app expects automatically.
  • Generic Credentials: Used heavily by the generic HTTP Request node. These include OAuth2, API Header Auth, Basic Auth, and Query Parameter Auth.

Adding Credentials: UI Walkthrough

We can create credentials in two ways:

  1. Global Panel: Click "Credentials" on the far-left sidebar → "Add Credential". This is best for setting up our environment before building anything.
  2. In-Line: While viewing a node's configuration panel, change Authentication to "Predefined Credential Type", choose our type (e.g., "Header Auth"), and click "Create New".
N8N_ENCRYPTION_KEY Importance: n8n encrypts our saved credentials in Postgres/SQLite using the N8N_ENCRYPTION_KEY. If our server dies and we restore our Postgres database but forget to copy over the exact N8N_ENCRYPTION_KEY from our old .env file, every single credential is permanently lost and must be re-entered manually. Treat this key like a cryptocurrency seed phrase.

Generic Authentication Types

1. API Key Auth

The vast majority of modern SaaS apps use an API Key. Sometimes it goes in a Header, sometimes in the Query URL.

  • Header Auth: We specify the Name (e.g., Authorization or X-Api-Key) and the Value (e.g., Bearer sk_live_12345).
  • Query Auth: We specify the Name (e.g., api_key) and the Value. n8n will automatically append ?api_key=12345 to the end of every HTTP Request that uses this credential.

2. Basic Auth

Legacy systems often use Basic Auth. We provide a User and Password. Under the hood, n8n automatically base64-encodes them and injects the Authorization: Basic {token} header for us.

3. OAuth2 (The Golden Child)

OAuth2 is notoriously difficult to code from scratch (exchanging authorization codes for access tokens, then writing cron jobs to refresh access tokens every 60 minutes). n8n turns OAuth2 into a 1-minute task.

To set up an OAuth2 Generic API credential:

  1. Create the credential and select "OAuth2 API".
  2. Copy the OAuth Redirect URL n8n generates for us (e.g., https://n8n.mycompany.com/rest/oauth2-credential/callback).
  3. Paste this exact URL into the external App's developer console (like Google Cloud Platform or Slack App settings).
  4. Copy the Client ID, Client Secret, Authorization URL, and Access Token URL from the external app into n8n.
  5. Click "Connect". n8n will open a popup, ask us to log in to the external service, and secure the token.

n8n will now handle the token refresh lifecycle in the background automatically forever.

Sharing Credentials: Once a credential is created, it is globally available. If we have 15 distinct workflows hitting the Google Sheets API, they all use the single "Google Sheets OAuth2" credential we created. If the connection breaks, we fix it in one place, and all 15 workflows instantly recover.

Real Example: Private API over HTTP

Assume we have an internal microservice running at http://internal-api:8000 that verifies license keys. It requires a Bearer token.

  1. Go to Credentials → Add Credential → Search "Header Auth".
  2. Name the credential: Internal API Token.
  3. Name: Authorization
  4. Value: Bearer super_secret_microservice_token_999
  5. Save.
  6. Open our workflow, drop an HTTP Request node.
  7. Set Authentication → Predefined Credential Type → Header Auth.
  8. Select "Internal API Token".
  9. Set the URL to http://internal-api:8000/verify.

n8n will inject the header beautifully. The actual Bearer token is completely hidden from anyone viewing the workflow canvas, preventing accidental screen-sharing leaks.

Real Example: Authenticate Google Sheets API

We want to read rows from a Google Sheet. Google enforces strict OAuth2.

  1. Go to Google Cloud Console → Create Project → Enable Google Sheets API.
  2. Go to Credentials → Create OAuth Client ID (Web Application).
  3. Inside n8n, drop a Google Sheets node onto the canvas.
  4. Change Authentication to OAuth2, click "Create New Credential".
  5. n8n will show us the exact OAuth Redirect URL. Copy it and paste it into the Google Cloud Console "Authorized redirect URIs" field.
  6. Copy Google's Client ID and Client Secret back into n8n.
  7. Click Connect my Account. Log in with our Google Workspace account and grant permissions.
  8. Select the exact Spreadsheet and Sheet Name in the node.

We can now drag and drop data into rows without writing a single line of Google's notoriously complex OAuth2 JWT refresh logic.
