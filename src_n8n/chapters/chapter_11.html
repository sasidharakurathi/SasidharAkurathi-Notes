The Array Mentality

In n8n, everything is an array of JSON objects (called "items"). When we fetch 50 users from a database, our node doesn't output one big JSON document with a users array. Instead, it outputs 50 distinct items. Every subsequent node we attach will run 50 times automatically.

This is incredibly powerful for simple 1-to-1 data moving, but what happens when we need to combine data from two different APIs? Or what if an API only accepts 10 items at a time? We use n8n's core data transformation nodes.

Merging Data Streams (The Join Operation)

The Merge node is how we stitch parallel execution paths back together. If our workflow splits-fetching data from Shopify on the top wire, and Postgres on the bottom wire-we use the Merge node to combine them.

  • Mode: Append. The simplest merge. If Shopify gives us 5 items, and Postgres gives us 3, the Merge node outputs 8 items sequentially.
  • Mode: Merge By Index. If we know for an absolute fact that item #1 from Shopify corresponds exactly to item #1 from Postgres, this mode zips them together horizontally into a single item.
  • Mode: Merge By Key. This is our equivalent to a SQL LEFT JOIN or INNER JOIN. We define the matching key (e.g., email). If Shopify item #2 has email: alice@test.com, and Postgres item #9 has email: alice@test.com, n8n merges their properties into one pristine item.
Merge Mode SQL Equivalent When to Use
Append UNION ALL Combining two independent lists into one stream
Merge by Index ZIP (positional) Both sources return identical item counts in matching order
Merge by Key LEFT JOIN / INNER JOIN Matching records across APIs by a shared field (email, ID)
⚠️ Warning: "Merge by Index" silently drops items if the two input streams have different lengths. If Shopify returns 5 items but Postgres returns 3, items #4 and #5 from Shopify will be lost. Always prefer "Merge by Key" when data counts might differ.

Processing Large Datasets: Split in Batches

Never pass 10,000 items into a single HTTP node loop if the target API has strict rate limits. We will get IP-banned immediately.

Instead, we use the Split in Batches node. We set the Batch Size (e.g., 50). n8n will take our 10,000 items, and release exactly 50 of them down the wire.

At the end of our processing chain, we attach the final node directly back to the Split in Batches node, creating a visual loop on the canvas. The loop automatically continues until all 10,000 items are exhausted.

Wait Nodes in Loops: When using Split in Batches to respect API rate limits, it is highly recommended to place a Wait node (e.g., Wait 2 Seconds) inside the execution loop. This acts as a throttle, ensuring we process exactly 50 items every 2 seconds without breaching quota ceilings.

Grouping Data: The Aggregate Node

If our workflow generated 15 individual items representing physical line-items in an order, and we need to POST one single JSON array containing all 15 items to an accounting API, we must use the Aggregate node.

The Aggregate node takes an array of individual items and squashes them into a single item containing an array field, essentially reversing n8n's default item expansion. We can also group by specific keys-calculating the sum of the amounts, the average value, or extracting an array of unique emails.

Sorting and Removing Duplicates

Data integrity is paramount when synchronizing CRMs or databases.

Sort Node: Does exactly what we expect. We define a field (e.g., createdAt) and ask it to sort ascending or descending. Crucially, it reorders the actual item indexing so the next node processes them chronologically.

Remove Duplicates Node: Before hitting a target API, we often want to strip out redundant data. Let's say we scraped 500 LinkedIn users, but 200 of them share the same company name. We can drop in a Remove Duplicates node, tell it to deduplicate based on the company_name field, and it will output exactly 300 unique items to the next node.

The Filter Node

Often, we do not want to use an IF node to split our workflow execution path. We simply want to aggressively prune our data stream.

The Filter node allows us to define complex conditional logic (e.g., transaction_status == 'PAID' AND amount > 500). Any item that matches the criteria continues to the next node. Any item that fails is instantly dropped from the execution entirely.

This is drastically cleaner than using an IF node and leaving the "False" path disconnected.

Real Example: SQL-style Joins Across APIs

Imagine we need to sync Intercom users with our internal Postgres database before updating a third dashboard.

  1. We add an Intercom node configured to fetch "All Users".
  2. We add a Postgres node configured to run SELECT id, email, ltv FROM users;
  3. We drag the outputs of both nodes into the inputs of a Merge node.
  4. We set the Merge Mode to Merge By Key.
  5. We specify the key name for Input 1 as email, and the key name for Input 2 as email.
  6. The Merge node outputs a beautiful, unified dataset containing Intercom's metadata merged horizontally with our database's LTV metrics. We can now pass this enriched data securely to our target dashboard.

Real Example: Processing 1000 Records in Batches

We receive a webhook containing 1000 new lead emails. We must enrich them using a tool like Clearbit, but Clearbit enforces a strict 60 requests-per-minute rate limit.

  1. Start with a Webhook node that outputs 1000 items.
  2. Connect a Split In Batches node. Set the Batch Size to 50.
  3. Connect an HTTP Request node (pointing to Clearbit). It will process the first 50 items.
  4. Connect a Wait node. Configure it to pause execution for 60 Seconds.
  5. Drag the output wire of the Wait node backward and drop it directly onto the input of the Split In Batches node.

The workflow will seamlessly loop: process 50 leads, sleep 60 seconds, grab the next 50 leads... until the entire webhook payload is processed safely without triggering DDoS protections.
