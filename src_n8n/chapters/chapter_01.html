Chapter 1: What is n8n? (The Mental Model)

Automation isn't new. But for a long time, the tools available fell into two frustrating camps: rigid, no-code SaaS platforms where we hit a wall the second we need custom logic, or writing custom microservices from scratch for every minor API integration. n8n sits explicitly in the middle.

n8n is a fair-code, node-based workflow automation tool built for developers. It gives us the visual canvas to map out API flows quickly, but hands us the escape hatch of raw JavaScript and direct HTTP requests when the built-in integrations aren't enough.

In Plain English: n8n is basically a visual orchestrator for API calls. Instead of writing a Node.js script that fetches from an API, parses JSON, and POSTs to another API, we drag nodes onto a canvas. But unlike other tools, if a node doesn't exist, we can just write a JavaScript snippet right in the workflow.

Why n8n For Developers?

Writing a custom Python or Node script to move data between two services takes an hour. Maintaining it forever takes weeks. We have to handle authentication, rate limiting, cron scheduling, webhook endpoints, error retries, and logging.

n8n handles the infrastructure of automation so we can focus only on the business logic. Here's what it brings to the table:

  • Self-hosting: We can run it on our own VPS (like Hetzner, DigitalOcean) via Docker. Our data never leaves our servers.
  • Code-Friendly: We get full access to the Code node where we can write raw JavaScript/TypeScript, use npm modules, and manipulate JSON data directly.
  • Transparent Data Flow: We can see the exact JSON payload moving from one node to another, making debugging incredibly fast compared to "magic" no-code tools.

Core Concepts

Before we build our first automation, let's nail down the fundamental vocabulary of n8n.

  • Workflow: The overall canvas. A collection of nodes connected together to perform a specific task.
  • Node: An individual step in a workflow. A node can be an app integration (like Slack or Postgres), a core logic node (like IF, Switch, or Code), or an HTTP Request. Every node takes JSON in and spits JSON out.
  • Trigger: The starting point of a workflow. Workflows don't run until a trigger fires. This can be a Schedule trigger (cron), a Webhook trigger, or an App trigger (e.g., "On new Row in Postgres").
  • Connection: The wire between nodes. Data in n8n flows sequentially from left to right along these connections.
  • Execution: A single run of a workflow. If a webhook fires 100 times, that creates 100 distinct executions, each with their own logs and data traces.
Developer Tip: Think of an n8n workflow as a functional pipeline. Each node is a function that receives an array of JSON objects (called Items), transforms them, and returns an array of JSON objects to the next node.

The n8n Architecture

When we deploy n8n (especially via Docker), we aren't just spinning up a single monolithic script. A production n8n setup consists of several distinct components:

  1. Editor UI: The frontend Vue.js application where we design workflows.
  2. Backend (Main Process): The Node.js server that serves the UI, manages the database (by default SQLite, but Postgres in production), and coordinates executions.
  3. Webhook Server (Optional but recommended): A separate process in high-volume environments dedicated only to listening for incoming HTTP webhooks so the main process doesn't get overwhelmed.
  4. Workers (Optional but recommended): In a scaled setup (Queue Mode using Redis), workers are separate containers that pick up and execute workflows, drastically improving performance.
Common Mistake: Do NOT use SQLite for a high-volume, production n8n instance. SQLite locks the database during writes, which causes executions to queue up and fail under heavy load. Always back our production n8n with PostgreSQL.

The Fair-code License Explained

n8n is often called "open source," but technically, it uses a fair-code license (specifically the Sustainable Use License).

What we CAN do:

  • Self-host n8n for free for our own internal company use.
  • Modify the source code.
  • Use it to automate processes for our clients as an agency.

What we CANNOT do:

  • We cannot offer n8n as a hosted service to third parties (we can't build "n8n-as-a-service" and sell access to the UI).

For 99% of developers building internal tools or client projects, n8n is effectively free and open.

When to use n8n vs Custom Scripts

n8n is powerful, but it isn't a silver bullet for every problem. Here's how we decide whether to map it out in n8n or open our IDE and write Python.

Use n8n when:

  • We need to stick APIs together quickly.
  • Visually debugging the data flow is helpful (e.g., complex, deeply nested JSON APIs).
  • We want built-in auth handling (OAuth2 flows are a nightmare to write from scratch; n8n handles token refreshes natively).
  • We need non-technical team members to be able to see or tweak the workflow later.

Write a custom script when:

  • We are doing heavy, CPU-bound processing (e.g., video rendering, massive matrix math, ML model training). n8n is an I/O wrapper, not a compute engine.
  • We need microsecond latency. n8n introduces overhead passing data between nodes.
  • We are managing millions of records in a single run. Passing gigantic datasets through node connections will cause Node.js out-of-memory crashes.
