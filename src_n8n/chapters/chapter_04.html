Chapter 4: Triggers - Starting a Workflow

The Manual Trigger

Every new workflow starts with a When clicking "Execute Workflow" node. This is the manual trigger. It does absolutely nothing in production. It strictly exists so we can test nodes interactively while building the workflow on the canvas.

When we are ready to deploy our logic, we will drop a real trigger onto the canvas and connect it to our first action node.

The Webhook Trigger (The Developer's Workhorse)

The Webhook node creates an instant HTTP endpoint on our n8n instance. When a third-party service makes an HTTP request to that URL, the workflow runs.

Webhook URL Structure:
https://{N8N_HOST}/webhook/{path} (Production)
https://{N8N_HOST}/webhook-test/{path} (Test)

Test vs Production URLs

n8n deeply embeds the concept of "Testing" vs "Production".

  • Test URL (/webhook-test/): Only listens when we explicitly click the "Listen for Test Event" button in the UI. Used for building and debugging. The event is captured on our canvas.
  • Production URL (/webhook/): Always listening in the background, 24/7 (as long as the workflow is "Active"). Does not show up on our canvas-we must view it in the Execution History tab.

Responding to Webhooks

By default, when a service hits our n8n webhook, n8n responds instantly with a 200 OK and a generic success message, and then runs the workflow. However, if we are building an API endpoint (e.g., a chatbot integration that requires a specific JSON response format), we can change the node setting "Respond" to "Using 'Respond to Webhook' Node". This stalls the HTTP response until n8n reaches a specialized response node later in our workflow, allowing us to return heavily processed data dynamically.

⚠️ Warning: The Production webhook URL (/webhook/) only works when the workflow is set to Active. If we forget to toggle the workflow on, external services will receive a 404 Not Found and silently fail.
Trigger Type How It Fires Latency Best For
Webhook External HTTP request hits n8n URL Instant (~50ms) Real-time events from GitHub, Stripe, Slack
Schedule (Cron) Internal timer fires at set intervals Depends on interval Batch jobs, daily reports, periodic sync
App Trigger (Instant) n8n auto-registers webhook with the app Instant Slack messages, GitHub events, Stripe payments
App Trigger (Polling) n8n checks the app's API periodically Minutes Legacy APIs without webhook support

The Schedule Trigger (Cron)

If we need to batch process data every night at 2 AM, or poll an old legacy API every 15 minutes, we use the Schedule trigger.

The node offers a simple UI for intervals (e.g., "Every an Hour"), but under the hood, developers can toggle "Pattern" mode to input raw Linux Crontab syntax.

# Run at 9:00 AM every day
0 9 * * *

# Run perfectly every 15 minutes
*/15 * * * *

# Run at 5:00 PM on Fridays only
0 17 * * 5
Developer Tip: Triple-check our GENERIC_TIMEZONE environment variable. If n8n boots in UTC, our "9 AM" cron job will suddenly trigger at 4 AM EST.

App-Specific Triggers

Instead of manually constructing webhook paths and mapping payloads, n8n provides pre-built trigger nodes for major platforms like GitHub, Stripe, Shopify, or Gmail.

When we drop a "GitHub Trigger" node, n8n asks for our GitHub credentials. When we activate the workflow, n8n automatically calls the GitHub API, creates the exact webhook configuration pointing to its own `/webhook` URL, and formats the incoming data cleanly. We never touch the raw repo settings.

Instant vs Polling Triggers

When using App Triggers, we must distinguish between the two structural types:

  • Instant (Webhooks): Event happens in the external app → App POSTs to n8n instantly. Lightning fast. No overhead.
  • Polling (Cron-based): The external app doesn't support webhooks (or n8n hasn't built them yet). n8n actively asks the app "Got any new data?" every X minutes.
Performance Warning: If we set a Polling trigger to check an API every 10 seconds, n8n creates a distinct execution record every 10 seconds, flooding our database. Be extremely conservative with polling intervals (aim for 10-15 minutes).

Real Example: Webhook Trigger (GitHub Push Event)

Let's say we have an internal dashboard and we want to log every time a developer pushes code to the main branch. We drop a raw Webhook Node onto the canvas.

  1. Set HTTP Method: POST
  2. Set Path: github-push-event
  3. Set Authentication: None (or basic, depending on setup)
  4. In GitHub Repo Settings → Webhooks, add our Production n8n URL: https://n8n.mycompany.com/webhook/github-push-event
  5. Set Content Type to application/json.

The instant GitHub pushes the payload, our n8n node fires. The payload is accessible in subsequent nodes like this:

// Accessing the pushed commit message and author
const commitMessage = $json.body.commits[0].message;
const pushedBy = $json.body.pusher.name;

// In a Slack node, we use the expression:
{{ $json.body.pusher.name }} just pushed: {{ $json.body.commits[0].message }}

Real Example: Cron Job (Daily 9 AM Report)

Imagine we need to email a daily Stripe sales report to the CEO every morning. We don't want to build a whole Python microservice with `schedule` loops for this.

  1. Add a Schedule Trigger node to the canvas.
  2. Set Rule: "Every Day"
  3. Set Time: "09:00"
  4. Add a Stripe node. Setup the action: Get All Charges (passing yesterday's timestamp).
  5. Add a Gmail or SendGrid node to email the output.

When the workflow is saved and set to "Active", n8n's background process adds this to its internal task queue. Tomorrow at exactly 9:00 AM server time, it pulls the data and sends the email entirely autonomously.
