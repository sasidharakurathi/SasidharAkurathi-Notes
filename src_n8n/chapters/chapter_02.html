n8n Cloud vs Self-Hosted

Before writing any workflows, we have to decide where the engine runs. n8n offers a managed SaaS offering (n8n Cloud) and a Self-Hosted option. For developers and engineers, the choice is usually obvious.

Decision Factor n8n Cloud Self-Hosted (Docker/VPS)
Setup Time 20 seconds 20 minutes (with Docker Compose)
Pricing Starts at ~€20/month (capped executions) Free (plus ~$5/mo for our Hetzner/DO VPS)
Limits Strict limits on execution counts Unlimited. Run 10 million workflows if our CPU can handle it.
Data Privacy Passes through n8n servers 100% Private. Ideal for internal DB connections or HIPAA/GDPR data.
Custom Modules Limited Install any npm package we want to use inside the Code node.

The 3 Ways to Self-Host

1. The "Just Let Me See It" Way (npx n8n)
If we have Node.js installed, open a terminal and run: npx n8n.
It downloads the package, spins up a local SQLite database, and opens the editor on port 5678. Never use this for production. It's just for quickly testing an idea on our laptop.
2. The "Default Developer" Way (Docker Run)
Running n8n in a standalone container.
docker run -it --rm --name n8n -p 5678:5678 -v n8n_data:/home/node/.n8n docker.n8n.io/n8nio/n8n
This is better because it isolates the environment, but it still relies on the built-in SQLite database by default, which creates locking issues if we run lots of workflows at once.
3. The "Production Standard" Way (Docker Compose)
This is how every serious n8n deployment is structured. We use a docker-compose.yml file to spin up n8n alongside PostgreSQL (to handle logs and execution data without locking) and an Nginx or Traefik container to handle HTTPS routing.

Environment Variables That Matter

n8n is almost entirely configured via environment variables. When setting up our .env file, we need to understand these five critical keys:

  • N8N_HOST: The domain name of our server (e.g., n8n.mycompany.com). Omit the protocol.
  • N8N_PORT: Internal port n8n listens on (default 5678).
  • WEBHOOK_URL: Crucial. This tells external services (like Stripe or Slack) where to send data back. It must be our full, public-facing URL: https://n8n.mycompany.com/.
  • N8N_ENCRYPTION_KEY: n8n stores credentials (API keys, DB passwords) in the database securely. It uses this key to encrypt them. Never lose this key. If we lose it, we lose access to every saved credential in our instance.
  • DB_TYPE: Set to postgresdb to rip out the default SQLite and point n8n at a robust database.
Common Mistake: If our webhooks generate URLs with localhost:5678 instead of our actual domain, it means we forgot to set or incorrectly formatted the WEBHOOK_URL variable.

Persistent Data Setup

If we don't bind a volume to our n8n Docker container, our workflows and credentials will vanish the second the server restarts. We must always map a volume to /home/node/.n8n.

However, when migrating to Postgres, n8n stores workflows and creds in Postgres instead of flat files. We still need a bound volume for binary files (like downloaded PDFs or images mid-workflow), but the DB handles the critical intelligence.

Reverse Proxy & SSL

n8n exposes an HTTP server on port 5678. We cannot expose port 5678 directly to the internet in a professional setup for two reasons:

  1. We need HTTPS (SSL) to encrypt our traffic (especially when sending API keys over the wire).
  2. Webhooks sent from external APIs expect standard ports (443).

We solve this with a Reverse Proxy like Nginx or Traefik. The proxy listens on port 443, handles the Let's Encrypt SSL certificate automatically, and securely routes the traffic internally to n8n's 5678 port.

Running as a Systemd Service?

If we aren't using Docker, some admins prefer installing n8n globally via npm and running it through a Linux systemd service or PM2. While valid, it heavily pollutes the host OS and makes migrating servers a nightmare. Always use Docker's restart: unless-stopped policy instead of relying on OS-level service managers.

Real Example: Production Docker Compose Stack

Here is the gold standard for deploying n8n on a fresh VPS. This stack includes n8n, PostgreSQL attached as the main database, and assumes we will use a reverse proxy (like Nginx) sitting in front of port 5678.

Developer Tip: Place this in a directory (e.g., /opt/n8n/), create a .env file next to it with our secrets, and run docker compose up -d. 
# docker-compose.yml
version: '3.8'

volumes:
  n8n_data:
  postgres_data:

services:
  postgres:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      - POSTGRES_USER=${DB_USER}
      - POSTGRES_PASSWORD=${DB_PASSWORD}
      - POSTGRES_DB=${DB_NAME}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      interval: 10s
      timeout: 5s
      retries: 5

  n8n:
    image: docker.n8n.io/n8nio/n8n
    restart: unless-stopped
    ports:
      - "127.0.0.1:5678:5678" # Only expose to localhost (proxy handles public traffic)
    environment:
      - N8N_HOST=${N8N_HOST}
      - N8N_PORT=5678
      - N8N_PROTOCOL=https
      - NODE_ENV=production
      - WEBHOOK_URL=https://${N8N_HOST}/
      - GENERIC_TIMEZONE=America/New_York
      - N8N_ENCRYPTION_KEY=${N8N_ENCRYPTION_KEY}
      
      # Tell n8n to use Postgres instead of SQLite
      - DB_TYPE=postgresdb
      - DB_POSTGRESDB_HOST=postgres
      - DB_POSTGRESDB_PORT=5432
      - DB_POSTGRESDB_DATABASE=${DB_NAME}
      - DB_POSTGRESDB_USER=${DB_USER}
      - DB_POSTGRESDB_PASSWORD=${DB_PASSWORD}
    volumes:
      - n8n_data:/home/node/.n8n
    depends_on:
      postgres:
        condition: service_healthy

By restricting n8n's port mapping to 127.0.0.1:5678, we ensure nobody can bypass the Nginx SSL proxy and hit the n8n application directly. All external traffic must go through port 443 via our proxy configuration.
