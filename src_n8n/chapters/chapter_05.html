The HTTP Request Node: The Swiss Army Knife

If n8n doesn't have an official integration for an app, it doesn't matter. The HTTP Request node is all we need. It gives us full control over raw REST API calls, supporting every standard HTTP method (GET, POST, PUT, PATCH, DELETE).

  • Auth Options: Built-in support for Basic, Header, Query Parameter, and OAuth2. We can configure OAuth2 directly in n8n credentials, and n8n will permanently handle the brutal token refresh lifecycle for us automatically.
  • Pagination Handling: Gone are the days of writing while (has_next_page) loops. Enable the "Pagination" setting on the node, define the condition (e.g., "Loop until response next_url is empty"), and n8n will automatically stitch all 500 pages of data into a single output Array.
  • Binary File Handling: Set "Response Format" to File. n8n will stream the PDF/Image into a strict binary format that we can immediately pipe into an AWS S3 or Google Drive node, completely avoiding RAM exhaustion.
Real Example: Fetching a massive list of paginated users from a CRM API. Set "Pagination" -> "Update Next URL" to expression {{ $response.body.links.next }}. n8n will fire 40 sequential HTTP requests and return 4,000 merged JSON items to the next node instantly.

The Code Node (JavaScript & Python)

When the UI nodes aren't enough, we write raw code. The Code node lets us execute custom JavaScript (or Python, if enabled in env variables) against the current execution data.

We access incoming data using the $input global object:

// 1. Getting Data
// Process all incoming items as an array
const items = $input.all();

// Process just the very first item
const firstItem = $input.first();

// 2. Modifying Data
for (const item of items) {
  // item.json holds the actual JSON payload
  item.json.fullName = `${item.json.firstName} ${item.json.lastName}`;
}

// 3. Returning Data
// Returning multiple items (fires the next node multiple times)
return items;

// Returning a single item (fires the next node once)
return [ items[0] ];
Warning on NPM Packages: By default, the Code node is sandboxed. We cannot require('axios') or import moment. To use external npm packages, we must explicitly whitelist them in our docker-compose file via the NODE_FUNCTION_ALLOW_EXTERNAL environment variable (e.g., NODE_FUNCTION_ALLOW_EXTERNAL=moment,lodash).

The Flow Nodes: Controlling Execution

Data manipulation is only half the battle. We need to control where the data goes.

The IF Node (Conditional Branching)

The IF node has two outputs: true and false. We define logical conditions (e.g., String containing "Urgent", Number greater than 100). If an item passes, it goes out the top. If it fails, it goes out the bottom.

Real Example: Filtering incoming emails. IF subject contains "Invoice", route to the Accounting workflow. Else, route to the General Inbox workflow.

The Switch Node (Multi-branch Routing)

When we have more than two outcomes, use Switch. We define rules (e.g., "Status = Pending", "Status = Paid", "Status = Failed"). n8n routes the specific JSON item down the matching wire. It functions identically to a classic switch...case statement.

The Merge Node (Combining Streams)

When we have a branched workflow that needs to reunite, or two separate data sources that need to be joined. The Merge node has multiple modes:

  • Append: Simply stacks Input 2's array at the bottom of Input 1's array.
  • Combine/Merge (SQL JOIN): Joins Item 1 from Input A with Item 1 from Input B based on a shared ID key. Incredible for marrying Stripe customer data with local Postgres user data.

The Loop Node (Loop Over Items)

By default, n8n runs nodes sequentially for every item. But sometimes we need to loop over items in batches (e.g., an external API only accepts 50 records per request). The Loop node takes 1,000 items, and outputs them in batches of 50. It creates a circular connection back to itself until the queue is empty.

The Utility Nodes

The Set Node (Edit Fields)

Instead of writing a 10-line Code node just to change a JSON key name or cast a string to an integer, use the Set node (often called "Edit Fields"). It allows us to forcefully set, rename, keep, or duplicate JSON values. If our webhook receives massive payloads but we only need the user_id and email, we use the Set node to keep only those two fields and discard the rest.

The Wait Node (Pausing Workflows)

Automation isn't always instant. The Wait node pauses the entire execution of the workflow. The execution goes to "sleep" in the database, consuming zero CPU.

  • Time-based: Wait exactly 3 days, or wait until next Tuesday at 9 AM.
  • Webhook-based: Pause the workflow indefinitely until a specific callback webhook URL is hit.

Real Example: Send an onboarding email. Wait 3 days. Send a follow-up email.

The Execute Workflow Node

The ultimate anti-spaghetti tool. Don't build a 200-node monster workflow. The Execute Workflow node allows us to trigger another n8n workflow as a sub-routine, pass our JSON data into it, and wait for it to return processed data.

Real Example: We have an "Email Parsing Engine" workflow. Instead of duplicating 30 email-processing nodes across 5 different automations, we just drop an Execute Workflow node in all of them, pointing to the single Engine workflow.

The Respond to Webhook Node

As mentioned in Chapter 4, this allows us to construct a custom HTTP response for an incoming webhook. We map the node to return a specific JSON payload, HTTP status code (like 400 Bad Request or 201 Created), and custom HTTP Headers.
