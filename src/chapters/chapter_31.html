Serving Full Web Pages

In Chapter 27, we learned how to return raw HTML strings using HTMLResponse. But writing an entire 500-line HTML website inside a Python string variable is an absolute nightmare to maintain.

If we want our FastAPI server to build and serve complex, dynamic websites (exactly like classic Django or Flask applications do), we use a templating engine. A templating engine allows us to write standard .html files and inject dynamic Python variables (like a user's name or a list of database items) directly into the HTML before sending it to the browser.

FastAPI natively supports Jinja2, which is the exact same industry-standard templating engine used by Flask. First, we must install it:

pip install jinja2

Setting Up Jinja2Templates

To start serving templates, we need to create a dedicated folder to hold all of our .html files. Let's create a folder named templates/ right next to our main.py.

Inside our Python code, we import Jinja2Templates and point it directly at that folder.

from fastapi import FastAPI, Request
from fastapi.responses import HTMLResponse
from fastapi.templating import Jinja2Templates

app = FastAPI()

# Tell FastAPI where our HTML files physically live
templates = Jinja2Templates(directory="templates")
In plain English: Jinja2Templates is a master object that scans our folder, reads our raw HTML files, dynamically injects our Python variables into them, and converts them into a final HTMLResponse for the browser.

Rendering a Template with Context

Let's create our first actual HTML file. Inside the templates/ directory, we create hello.html. 

<!-- templates/hello.html -->
<html>
<head>
    <title>Hello Page</title>
</head>
<body>
    <h1>Welcome to our startup!</h1>
    
    <!-- This is Jinja2 syntax! We inject a variable here -->
    <p>Hello, {{ username }}! Your ID is {{ id }}.</p>
</body>
</html>

Now, let's write the FastAPI route that renders this file. To render a template, our route absolutely must accept the Request object as a parameter, and pass it into the template renderer. 

# main.py
@app.get("/users/{id}", response_class=HTMLResponse)
async def read_user(request: Request, id: str):
    # Fetch a fake user from a database
    real_username = "Alice" 

    # We use TemplateResponse to stitch it all together!
    return templates.TemplateResponse(
        request=request, 
        name="hello.html", 
        context={"id": id, "username": real_username}
    )

Lines 7-11: The magic happens here.

  • request=request: Jinja2 requires the raw HTTP request object to function properly.
  • name="hello.html": The exact filename inside our templates/ folder.
  • context={...}: A dictionary of all the Python variables we want to aggressively inject into the HTML file. The keys in this dictionary must exactly match the {{ variables }} enclosed in double-brackets inside the HTML file!

Combining Templates with Static Files

A bare HTML page is ugly. We want to add beautiful CSS and maybe some JavaScript. To do this, we combine the Jinja2 templates we just learned with the StaticFiles we learned about in Chapter 25.

First, we mount our static folder in main.py:

from fastapi.staticfiles import StaticFiles

# Mount a "static" folder to hold our styles.css
app.mount("/static", StaticFiles(directory="static"), name="static")

Now, inside our hello.html template, how do we link to styles.css? We could hardcode the path like <link href="/static/styles.css">, but what if we change our server configuration later? The link will break.

Instead, Jinja2 provides a brilliant built-in function called url_for(). We can use it directly inside our HTML to dynamically generate the exact, perfectly safe URL for our static files.

<!-- templates/hello.html -->
<html>
<head>
    <title>Beautiful Page</title>
    
    <!-- Dynamically link our CSS file! -->
    <link href="{{ url_for('static', path='/styles.css') }}" rel="stylesheet">
</head>
<body>
    <h1 class="fancy-title">Hello, {{ username }}!</h1>
</body>
</html>
In plain English: Combining Jinja2Templates with StaticFiles allows us to build and host exactly the same kind of massive, beautiful, server-rendered websites as Django or Ruby on Rails, while keeping our code incredibly fast and lightweight.
