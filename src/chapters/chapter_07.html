Taking Control with Field()

Till now we briefly saw how to use Field() to add extra validation to our Pydantic models. Now let's explore its true power. Think of Field() as a strict bouncer who has a very detailed set of instructions.

from pydantic import BaseModel, Field

class Item(BaseModel):
    name: str = Field(min_length=2, max_length=50)
    description: str = Field(
        default=None, 
        title="Item Description", 
        description="A detailed description of the item"
    )
    price: float = Field(gt=0, le=1000)
    tax: float = Field(default=0.0, ge=0)

Here are the most useful options we can pass to Field():

  • default: The default value if the user does not provide one.
  • title & description: These show up beautifully in our interactive Swagger UI docs.
  • min_length & max_length: For strings. Prevents names that are too short or essays where just a short name is expected.
  • gt, ge, lt, le: For numbers. Greater than, greater or equal, less than, less or equal.
  • pattern: A regular expression that the string must perfectly match.

Nested Models (Models inside Models)

Real-world data is rarely flat. When ordering a product online, an order is not just a list of text. An order contains an item, and an item has its own details. We can represent this deep structure by putting Pydantic models inside other Pydantic models.

class Image(BaseModel):
    url: str
    name: str

class Item(BaseModel):
    name: str
    description: str = None
    image: Image = None

Lines 1-3: We create a standard model for an Image.

Line 8: We use our Image model as the type for the image attribute inside our Item model. Now, our Item perfectly understands that it contains another complex object inside of it.

In plain English: We can snap Pydantic models together like Lego blocks to build deeply layered, perfectly validated data structures.

Lists of Nested Models

An item usually has more than one image. What if we want a list of images? We just wrap our nested model in the standard Python list (or List from typing).

from typing import List

class Item(BaseModel):
    name: str
    description: str = None
    images: List[Image] = None

If a user tries to send a list containing three perfect images and one broken image, FastAPI will stop the entire request and tell the user exactly which image in the list is broken!

Set Types for Unique Items

Sometimes, a list is not enough. What if we want to ensure every image in the list is absolutely unique? We do not want users uploading the exact same image URL five times.

To enforce uniqueness automatically, we use the Python set type instead of list.

from typing import Set

class Item(BaseModel):
    name: str
    images: Set[Image] = set()

Even if the user sends a request with duplicate items in the JSON array, FastAPI will automatically filter out the duplicates before our code even runs.

Deeply Nested Models

There is no limit to how deep we can go. We can have an Offer model that contains a list of Item models, and each Item model contains a list of Image models. 

class Offer(BaseModel):
    name: str
    description: str = None
    price: float
    items: List[Item]

FastAPI will validate this entire massive tree of data in a fraction of a millisecond. If we tried to write this validation manually with loops and if-statements, it would take hundreds of lines of code.

⚠️ Common Mistake: Defining models in the wrong order. Because Python reads top-to-bottom, we must define the "inner" models (like Image) before we try to use them inside the "outer" models (like Item).

Special Types: URLs and Emails

Validating an email address is famously difficult. We could write a massive Regex pattern, but FastAPI provides built-in special types that handle this for us automatically.

By installing an extra package (pip install "pydantic[email]"), we unlock magical types like EmailStr and HttpUrl. 

from pydantic import BaseModel, HttpUrl, EmailStr

class UserProfile(BaseModel):
    name: str
    email: EmailStr
    website: HttpUrl

If the user sends "not-an-email" for the email, or "www.bad-link" (without the http://) for the website, Pydantic immediately rejects the request with a detailed error. We get incredibly powerful validation with zero effort.

Bodies as Dictionaries

Sometimes, we want to accept a dictionary where we know the type of the keys and the values, but we don't know their exact names ahead of time. Think of recording temperature readings from dozens of differently-named sensors.

from typing import Dict

@app.post("/index-weights/")
def create_weights(weights: Dict[int, float]):
    return weights

This tells FastAPI to expect a JSON object where every key is an integer, and every value is a float (like {"1": 4.5, "2": 9.1}).

Union Types (Accepting Multiple Options)

What if a field could be an integer OR a string? We use the Union type from Python to allow multiple valid options.

from typing import Union

class Item(BaseModel):
    name: str
    specs: Union[str, int] = None

FastAPI will first check if the incoming data is a string. If it is not, it will check if it is an integer. If both fail, it returns an error.

Using Annotated with Field()

As we use modern Python, we will often see Annotated used alongside Field(). It is a cleaner way to attach validation rules to our types, especially when reusing them.

from typing import Annotated
from pydantic import BaseModel, Field

class Item(BaseModel):
    name: str
    price: Annotated[float, Field(gt=0, description="The cost")]

This reads as: "The price is a float, and by the way, it must be greater than zero." It separates the data type from the validation rules beautifully.
