CHAPTER 6 - Request Bodies

What is a Request Body?

Until now, we have only sent tiny pieces of data to our API through the URL (like an ID or a search keyword). But what happens when we need to send a massive amount of data? Imagine filling out a long registration form with our name, password, address, and a profile picture.

We cannot fit all of that into a URL. Instead, we bundle all that data into a secure package and send it behind the scenes. This hidden package of data is called the Request Body.

In plain English: A Request Body is a block of data sent from the browser to the API. We use it when we have too much data to fit in a URL, or when the data is sensitive (like passwords).
Use this when: We are creating new database records (POST) or updating existing ones (PUT/PATCH).

Don't use this when: We are just making a simple GET request to fetch data. GET requests should almost never have a body!

Our First Pydantic Model

To accept a request body in FastAPI, we need to tell it exactly what data to expect. We do this by creating a blueprint using a tool called Pydantic.

Think of a Pydantic model like a strict digital form. It lists all the required fields, and the type of data (text, numbers) each field accepts.

from fastapi import FastAPI
from pydantic import BaseModel

class Item(BaseModel):
    name: str
    description: str = None
    price: float
    tax: float = None

app = FastAPI()

Let's break this down:

Line 2: We import BaseModel from Pydantic. This gives us the superpowers needed to create our blueprint.

Line 4: We create our blueprint by defining a Python class that inherits from BaseModel.

Lines 5-8: We define our fields. Notice how name and price don't have default values? That means they are strictly required. But description and tax equal None, making them completely optional.

Using the Model as a Body Parameter

Now that we have our strict blueprint, all we have to do is drop it into our path operation function.

@app.post("/items/")
def create_item(item: Item):
    return item

Line 1: We use @app.post because we are receiving data to create something new.

Line 2: We add item to our function, and declare its type as Item (the Pydantic class we just built).

Because Item is a Pydantic model, FastAPI instantly knows this data must come from the Request Body, not the URL!

⚠️ Common Mistake: Forgetting to use POST. If we try to send a body using @app.get, many browsers and servers will block the request entirely.

Accessing Individual Fields

When the request body arrives successfully, FastAPI hands us a fully-formed Python object. We don't have to parse messy JSON or dictionaries manually. We can just use standard Python "dot" notation to access the inner fields. 

@app.post("/items/")
def create_item(item: Item):
    item_dict = item.model_dump()
    if item.tax:
        price_with_tax = item.price + item.tax
        item_dict.update({"price_with_tax": price_with_tax})
    return item_dict

Line 3: We use item.model_dump() (in older FastAPI versions this was .dict()) to convert our strict object back into a standard Python dictionary.

Line 4 & 5: We easily access the inner fields just by typing item.tax and item.price.

Path + Query + Body: The Ultimate Combo

FastAPI is smart enough to handle all three types of inputs simultaneously. We just drop them all into our function, and FastAPI automatically sorts them out based on our type hints.

@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item, q: str = None):
    result = {"item_id": item_id, **item.model_dump()}
    if q:
        result.update({"q": q})
    return result

Look at how brilliantly FastAPI routes this:

  • item_id is an integer, and it matches the {item_id} in the path. FastAPI grabs it from the URL path.
  • item is a Pydantic model. FastApi grabs it from the secure Request Body.
  • q is a string, and it is NOT in the path. FastAPI grabs it from the Query Parameters (the ?q= part of the URL).

Extra Validation with Field()

Just like we used Path() and Query() previously, we can use Field() to add strict mathematical or length validation directly inside our Pydantic models.

from pydantic import BaseModel, Field

class Item(BaseModel):
    name: str
    description: str = Field(
        default=None, title="The description", max_length=300
    )
    price: float = Field(gt=0, description="The price must be greater than zero")

Lines 5-8: We import Field from Pydantic. We then use it to ensure the description cannot exceed 300 characters, and the price is strictly greater than zero.

Body with Multiple Models

What if a user is checking out of our shopping cart, and they need to send us their Item details AND their User details at the exact same time?

We do not have to merge everything into one giant model. We can just add both models to our function.

class User(BaseModel):
    username: str
    full_name: str = None

@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item, user: User):
    return {"item_id": item_id, "item": item, "user": user}

Line 6: By adding both item: Item and user: User, FastAPI expects the JSON request body to contain two distinct sections: one labeled "item" and one labeled "user". It manages this complexity automatically.

Singular Values in Body using Body()

Sometimes, we want to receive a single, simple value in the request body (like an "importance" score), but it feels silly to create an entire Pydantic BaseModel class just for one single integer.

If we just add importance: int to our function, FastAPI will think it is a Query Parameter. To force FastAPI to look inside the body for this single value, we use the Body() tool.

from fastapi import Body

@app.put("/items/{item_id}")
def update_item(
    item_id: int, item: Item, user: User, importance: int = Body()
):
    return {"item_id": item_id, "item": item, "user": user, "importance": importance}

Line 5: By setting importance = Body(), FastAPI knows to dig into the JSON request body to find this integer, rather than looking at the URL.

Embedding a Single Body Parameter

When we only have one Pydantic model (like Item), FastAPI expects the JSON body to look like a flat dictionary consisting of the item's details. It expects {"name": "Foo", "price": 45.2}.

But sometimes we want the JSON to explicitly name the model, like this: {"item": {"name": "Foo", "price": 45.2}}.

We can force FastAPI to wrap the data in this named outer shell by using Body(embed=True).

@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item = Body(embed=True)):
    return {"item_id": item_id, "item": item}

Line 2: By passing embed=True, FastAPI demands that the JSON body must have an outer key named "item". This is extremely useful if we plan to add more models to this route in the future and want to keep our JSON structure consistent.

In plain English: The Request Body is where the heavy lifting happens. With Pydantic, we create strict blueprints that automatically read, catch typing errors, and structure giant blocks of data inside our Python functions securely.
