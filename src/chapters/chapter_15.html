Raising HTTPException

In Chapter 12, we learned about HTTP status codes. But what happens when something actually goes wrong inside our code? Maybe a user asks for an item ID that doesn't exist in our database, or they try to access a page they don't have permission to see.

We cannot just return a normal Python dictionary with an error message, because FastAPI will wrap that in a happy "200 OK" status code. Instead, we must strictly halt our code and throw an error back to the browser. We do this by raising an HTTPException.

from fastapi import FastAPI, HTTPException

app = FastAPI()

items = {"foo": "The Foo Wrestlers"}

@app.get("/items/{item_id}")
async def read_item(item_id: str):
    if item_id not in items:
        raise HTTPException(status_code=404, detail="Item not found")
    return {"item": items[item_id]}

Line 10: Notice we use the Python keyword raise, not return. Raising an HTTPException instantly stops our function right there. The code below it never runs. FastAPI catches the exception, creates a clean JSON response with the detail message, and sets the exact HTTP status code we asked for (like a 404 Not Found).

In plain English: Use raise HTTPException() whenever we need to immediately stop our code, send an error message, and return a failure status code (like 400 or 404).

Custom Error Details

The detail parameter of an HTTPException doesn't have to be a simple string. Depending on how complex our error is, we can pass a Python list or even a full dict (dictionary). FastAPI will cleanly convert whatever we pass into beautiful JSON.

@app.get("/users/{user_id}")
async def read_user(user_id: int):
    if user_id > 100:
        raise HTTPException(
            status_code=400, 
            detail={"error": "ID too high", "max_allowed": 100}
        )
    return {"user": "Valid User"}

Custom Exception Handlers

What if our application relies on a totally custom Python error? Let's say we wrote a completely unique UnicornException that gets triggered whenever a magical database goes down.

If we just raise UnicornException(name="Sparkles") in our routing code, FastAPI won't know what to do with it, and it will return an ugly "500 Internal Server Error" to the user.

We can solve this globally! We can teach FastAPI how to handle our custom exceptions automatically using the @app.exception_handler decorator. 

from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse

class UnicornException(Exception):
    def __init__(self, name: str):
        self.name = name

app = FastAPI()

@app.exception_handler(UnicornException)
async def unicorn_exception_handler(request: Request, exc: UnicornException):
    return JSONResponse(
        status_code=418,
        content={"message": f"Oops! {exc.name} did something. There goes a rainbow..."},
    )

@app.get("/unicorns/{name}")
async def read_unicorn(name: str):
    if name == "yolo":
        raise UnicornException(name=name)
    return {"unicorn_name": name}

Let's unpack this magic:

Lines 4-6: We create a standard Python exception class. Nothing related to FastAPI yet.

Line 10: We use @app.exception_handler and pass our custom class. We are telling FastAPI: "If you ever see a UnicornException anywhere in our app, stop what you are doing and run this function instead."

Lines 12-15: Inside the handler, we must manually return a JSONResponse, picking exactly what status code (like the joke 418 I'm a teapot) and what content to send back.

Line 20: Inside our actual route, we just cleanly raise UnicornException. The global handler catches it perfectly.

Use this when: We have core database or logic exceptions that might be thrown from deep inside our code, and we want to reliably catch them at the top level to return clean JSON errors to the user.

Overriding Default Handlers

FastAPI uses Pydantic to validate data. When Pydantic catches bad data (like an integer instead of a string), FastAPI throws an internal error called RequestValidationError and returns a 422 Unprocessable Entity status.

But what if we don't like the way FastAPI's default 422 errors look? What if our frontend application expects error messages in a very specific format? We can override FastAPI's built-in handlers using the exact same trick!

from fastapi import FastAPI, Request, status
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse

app = FastAPI()

@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request: Request, exc: RequestValidationError):
    return JSONResponse(
        status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
        content={
            "error_type": "VALIDATION_FAILURE",
            "messy_details": exc.errors(),
            "body_received": exc.body
        },
    )

Line 7: By putting RequestValidationError into our exception handler, we completely hijack FastAPI's default validation error process.

Line 13: The exc.errors() function gives us the raw list of exactly what Pydantic complained about.

Line 14: The exc.body gives us the exact raw JSON the user tried to send us, which is incredible for debugging.

Overriding the HTTPException Itself

We can even override FastAPI's standard HTTPException globally if we wanted to enforce a strict corporate response format (for example, wrapping all errors in an "error_envelope": {} dictionary).

The only trick here is ensuring we re-use FastAPI's default handlers if we only want to tweak a tiny thing. We can import them via from fastapi.exception_handlers import http_exception_handler and run them inside our custom overrides!
