CHAPTER 26 - SQL Databases

SQLAlchemy Explained Simply

Until now, every time we restarted our server, our fake dictionary database was completely erased. To build real applications, we need a real database (like PostgreSQL, MySQL, or SQLite). However, writing raw SQL queries (like SELECT * FROM users) natively inside Python is incredibly messy and prone to security hacks.

Instead, the professional Python world uses an ORM (Object-Relational Mapper). An ORM translates normal Python classes directly into SQL tables automatically.

SQLAlchemy is the undisputed king of Python ORMs. It is incredibly powerful, but notoriously complex. Because of this, the creator of FastAPI built a sister-library called SQLModel. SQLModel is literally just SQLAlchemy and Pydantic perfectly merged together to make our lives infinitely easier.

In plain English: SQLAlchemy handles the heavy database lifting. SQLModel wraps SQLAlchemy in a beautiful, simple syntax specifically designed for FastAPI. We will cover both approaches!

The Crucial Concept: Two Types of Models

Before we write any code, we must understand the most critical distinction in all of FastAPI database design:

  • The Pydantic Schema: This defines the shape of the JSON that goes in and out of our API over the internet. (e.g., A user sends a password to register).
  • The ORM Model: This defines the shape of the data stored inside the database. (e.g., We never store the plain password, we store a hashed_password).

If we use pure SQLAlchemy, these two models are written completely separately. If we use SQLModel, we can brilliantly combine them into one seamless class!

Setting Up the Engine (Both Methods)

Whether we use SQLAlchemy or SQLModel, we always need an "Engine". The engine is the connection between our Python code and the database.

# database.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, declarative_base

# We use SQLite for an easy local file database
SQLALCHEMY_DATABASE_URL = "sqlite:///./sql_app.db"

# 1. Create the engine
engine = create_engine(
    SQLALCHEMY_DATABASE_URL, connect_args={"check_same_thread": False}
)

# 2. Create a Session Factory
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

# 3. Create the Base Class (For pure SQLAlchemy)
Base = declarative_base()

Defining the Models: Pure SQLAlchemy

If we use raw SQLAlchemy, we must inherit from the Base we just created, and manually map out exactly what the SQL table looks like.

# models.py (SQLAlchemy ORM Models)
from sqlalchemy import Boolean, Column, Integer, String
from database import Base

class User(Base):
    __tablename__ = "users"

    id = Column(Integer, primary_key=True, index=True)
    email = Column(String, unique=True, index=True)
    hashed_password = Column(String)
    is_active = Column(Boolean, default=True)

Notice we had to use Column() everywhere. To make our API work, we now have to go build a completely duplicate Pydantic schema in schemas.py just to accept the JSON payload.

Defining the Models: SQLModel (The Better Way)

Let's look at how much cleaner this is using SQLModel. We don't need a Base class, we just inherit from SQLModel. It automatically builds both the database table and the Pydantic JSON schema at the exact same time!

# models.py (SQLModel)
from typing import Optional
from sqlmodel import Field, SQLModel

class User(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    email: str = Field(unique=True, index=True)
    hashed_password: str
    is_active: bool = True

That's it! Because we added table=True, SQLModel knows to build a real database table. But under the hood, this is a fully functioning Pydantic model we can use directly in our routes.

The Database Session Dependency

Now that our database is configured, how do our routes actually query it? We need a Session. A session is a temporary workspace; we fetch data, make changes, and then close the workspace.

We do not want to accidentally leave sessions open forever, so we use a dependency with the yield keyword.

# dependencies.py
from database import SessionLocal

def get_db():
    db = SessionLocal()
    try:
        # Give the route the database connection
        yield db 
    finally:
        # Ensure we always close it when the route finishes!
        db.close()

CRUD Operations: SQLAlchemy Example

CRUD stands for Create, Read, Update, Delete. Let's see how we use our get_db dependency to perform a "Read" and a "Create" using the pure SQLAlchemy approach.

from fastapi import FastAPI, Depends, HTTPException
from sqlalchemy.orm import Session
from database import get_db
import models, schemas

app = FastAPI()

# Read (Fetch a user by ID)
@app.get("/users/{user_id}", response_model=schemas.User)
def read_user(user_id: int, db: Session = Depends(get_db)):
    # Run the raw SQLAlchemy query
    db_user = db.query(models.User).filter(models.User.id == user_id).first()
    if db_user is None:
        raise HTTPException(status_code=404, detail="User not found")
    return db_user

# Create (Insert a new user)
@app.post("/users/", response_model=schemas.User)
def create_user(user: schemas.UserCreate, db: Session = Depends(get_db)):
    # 1. We receive the clean Pydantic JSON schema
    # 2. We convert it into the SQLAlchemy ORM model
    fake_hashed_password = user.password + "notreallyhashed"
    db_user = models.User(email=user.email, hashed_password=fake_hashed_password)
    
    # 3. We add it to the session and commit (save) it to the real database!
    db.add(db_user)
    db.commit()
    db.refresh(db_user)
    return db_user

CRUD Operations: SQLModel Example

If we use SQLModel, the process is incredibly similar, but the queries themselves look much more like modern Python.

from sqlmodel import Session, select

# Read with SQLModel
@app.get("/users/{user_id}")
def read_user_sqlmodel(user_id: int, db: Session = Depends(get_db)):
    # SQLModel uses a cleaner 'select' syntax
    statement = select(User).where(User.id == user_id)
    user = db.exec(statement).first()
    if not user:
        raise HTTPException(status_code=404, detail="User not found")
    return user
Which should we choose?
If we are joining an existing monolithic company enterprise, they are already using raw SQLAlchemy. It is the industry standard.

If we are building a brand new FastAPI application from scratch today, we should absolutely use SQLModel. It eliminates massive amounts of duplicate code while keeping all the power of SQLAlchemy under the hood.
