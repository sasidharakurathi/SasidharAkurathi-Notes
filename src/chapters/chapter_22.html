CHAPTER 22 - APIRouter (Splitting Your App)

Why We Need Routers

So far, we have built every single route directly on our main app object using things like @app.get(). This works brilliantly for small projects. But what happens when our startup grows?

Imagine we have 20 endpoints for managing users, 30 for handling products, and 15 for processing payments. If we write 65 endpoints in a single main.py file, it becomes a massive, unreadable nightmare of thousands of lines of code.

Think of our API like a massive company. The CEO (our main.py file) cannot personally handle every single task in the entire company. Instead, the CEO delegates work to different departments: HR, Sales, and Accounting. Each department manages its own specialized tasks.

In FastAPI, these "departments" are called APIRouters.

In plain English: APIRouter is a "mini FastAPI application." It allows us to split our codebase into multiple files, build routes separately, and plug them all into our main app at the very end.

Creating Our First APIRouter

Let's create a brand new file called users.py to handle all of our user logic. Instead of importing FastAPI, we import APIRouter. 

# users.py
from fastapi import APIRouter

# This is our mini-app for the "Users" department
router = APIRouter()

@router.get("/users/")
async def read_users():
    return [{"username": "johndoe"}, {"username": "alice"}]

@router.get("/users/{username}")
async def read_user(username: str):
    return {"username": username}

Line 5: We instantiate the APIRouter(). From this point on, we use @router instead of @app to build our paths.

Prefix, Tags, and Responses

If we look closely at our users.py file, we notice something repetitive. Every single route starts with /users/. Furthermore, if we want these routes to be grouped beautifully under a "Users" heading in our Swagger documentation, we would have to add tags=["Users"] to every single @router.get() definition.

APIRouter lets us define these common elements one single time when we create the router!

# users.py
from fastapi import APIRouter, HTTPException

router = APIRouter(
    prefix="/users",
    tags=["Users"],
    dependencies=[Depends(verify_token)],
    responses={404: {"description": "User not found"}},
)

@router.get("/")
async def read_users():
    return [{"username": "johndoe"}]

@router.get("/{username}")
async def read_user(username: str):
    return {"username": username}

Lines 4-9: We configured the entire department at once!

  • The prefix="/users" means our route on Line 11 (@router.get("/")) magically becomes /users/. The route on Line 15 becomes /users/{username}. (Note: The prefix cannot end in a slash).
  • The tags=["Users"] cleanly groups all of these routes in our Swagger docs.
  • The responses dictionary ensures every route automatically inherits that 404 documentation.

Router-Level vs Route-Level Dependencies

The most powerful feature of the APIRouter is the dependencies list on Line 7. Imagine we want every single route in the Users department to require a valid security token.

Instead of manually typing Depends(verify_token) on 20 different endpoints (route-level dependencies), we inject it once into the APIRouter itself (router-level dependency). FastAPI will aggressively enforce that dependency on every single path inside that router!

Including a Router in the Main App

Right now, our users.py router is completely disconnected. We have built the HR department, but the CEO doesn't know it exists. We must plug the router into our main.py application.

# main.py
from fastapi import FastAPI
import users
import products # Imagine we also built a products.py!

app = FastAPI()

# Plug the "departments" into the main company
app.include_router(users.router)
app.include_router(products.router)

@app.get("/")
async def root():
    return {"message": "Welcome to our massive API!"}

Lines 9-10: The app.include_router() method is the glue. It sweeps through the imported users.py file, gathers up all the routes attached to users.router, prepends any prefixes, and bolts them permanently onto our main FastAPI application.

In plain English: We use main.py exclusively to configure the master app and plug in all our include_router statements. We write the actual logic in separate files using APIRouter().

Nesting Routers

In massive, enterprise-grade applications, we can even plug routers into other routers! We could have an admin_router, and we could include_router(users.router) directly into the admin_router (which might add another prefix like /admin/users). This allows for incredibly modular, highly organized codebases.
