Why Hardcoding is Bad

Imagine we are building our startup's API, and we have a critical database password: super_secret_db_pass_123. If we copy and paste that exact string directly into our database.py file, we have created a massive security and operational nightmare.

First, if we ever upload our code to GitHub, the entire internet can instantly see our password and steal our company's data. Secondly, when we finally launch our app, our production server will need a completely different password than our local development laptop. We cannot manually rewrite the code every single time we switch computers.

The universal solution in software engineering is to use Environment Variables. These are invisible variables that live directly on the operating system of the server itself, totally separated from our code.

Pydantic Settings: The Modern Approach

Traditionally, developers used os.getenv("DATABASE_URL") to fetch these variables from the server. But FastAPI was built by the creator of Pydantic, so we have something vastly superior: BaseSettings.

First, we need to install the companion package: pip install pydantic-settings.

from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    app_name: str = "My Awesome API"
    admin_email: str
    items_per_user: int = 50
    debug_mode: bool = False

# We instantiate grabbing the server's environment variables instantly!
settings = Settings()

This is incredibly powerful! Pydantic will automatically look at the operating system's environment variables. If it sees a variable called ADMIN_EMAIL, it will automatically pull it in and attach it to settings.admin_email.

In plain English: BaseSettings takes messy, unreliable operating system text strings and transforms them into perfectly safe, validated, and auto-completed Python objects.

Type-Safe Configuration

The absolute greatest feature of Pydantic BaseSettings is that it guarantees Type Safety.

Operating system environment variables are always strings. If we set an environment variable locally like ITEMS_PER_USER="50" or DEBUG_MODE="False", the raw OS just sees quotes. If we used standard Python os.getenv(), we would have to manually parse those strings into integers and booleans.

But Pydantic automatically reads our type hints (int, bool)! When it reads the string "50", it instantly converts it to an integer. When it reads the string "False" (or "0" or "off"), it instantly converts it to a boolean False!

If we accidentally set the server's environment variable to ITEMS_PER_USER="apple", our FastAPI server will instantly crash on startup and scream at us with a beautiful validation error, completely preventing a disaster in production.

.env File Support

Setting raw environment variables directly on our Windows or Mac laptops every time we want to code is extremely annoying. Instead, we usually create a hidden text file named literally just .env in the root of our project.

# .env (NEVER upload this file to GitHub!)
ADMIN_EMAIL=ceo@ourstartup.com
ITEMS_PER_USER=100
DEBUG_MODE=True
SECRET_API_KEY=x9f8ja83n2b

By simply adding model_config to our Settings class, Pydantic will automatically scan our folder, find the .env file, and load all the variables as if they were real server variables. 

from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    admin_email: str
    items_per_user: int = 50
    debug_mode: bool = False
    secret_api_key: str

    # Tell Pydantic to read our hidden file!
    model_config = SettingsConfigDict(env_file=".env")

Different Configurations for Dev vs Prod

Because Pydantic is so flexible, we easily manage the difference between our local laptop (Development) and our remote company server (Production).

On our local laptop, we create a .env file holding our localhost database URLs and fake passwords. When we write our code, everything runs against the local tests.

When we upload our code to our production server (like AWS or render), we absolutely do not include the .env file. Instead, we manually type real, secure Environment Variables directly into the AWS dashboard.

When our FastAPI server boots up in production, Pydantic looks for the .env file, fails to find it (which is perfectly fine), and seamlessly falls back to reading the true Environment Variables from the AWS server itself!

A Full Working Settings Example

Let's look at how we inject our beautifully typed settings into our actual routes using the magical lru_cache. We don't want Python to read from the hard drive 1,000 times a second, so we cache the settings object! 

# config.py
from functools import lru_cache
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    app_name: str = "My Awesome API"
    admin_email: str
    database_url: str

    model_config = SettingsConfigDict(env_file=".env")

@lru_cache
def get_settings():
    # Only runs once per startup!
    return Settings()
  
# main.py
from fastapi import FastAPI, Depends
from config import get_settings, Settings

app = FastAPI()

@app.get("/info")
async def info(settings: Settings = Depends(get_settings)):
    return {
        "app_name": settings.app_name,
        "admin_contact": settings.admin_email,
        # We successfully used our settings dynamically!
    }
