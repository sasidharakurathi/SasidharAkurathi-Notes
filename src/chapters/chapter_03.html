CHAPTER 3 - Path Operations (The Building Blocks)

What is a "Path Operation"?

Think of an API like a massive vending machine. The "path" is the exact button we press. The "operation" is what action we are asking the machine to do (buy a snack, return change, restock). In FastAPI, a Path Operation is the combination of an exact URL path and a specific HTTP method.

Before any user can interact with our application, we must define exactly which paths are available, and what actions are allowed on those paths.

In plain English: A path operation tells FastAPI: "When someone visits this specific URL, and they use this specific HTTP method, run this exact Python function."

The Core HTTP Methods

To talk to a web server, our browser does not just send a URL. It also sends a "method" to explain its intention. Most of the time, our browser just asks to read data, but APIs allow users to do much more.

Here is a guide to the methods we will use in FastAPI, and when to use them:

Method Real World Analogy Best Used For
GET Reading a book at the library. Reading or fetching data. We are not changing anything.
POST Dropping a letter in a mailbox. Creating brand new data, like registering a new user.
PUT Replacing a broken car engine with a new one. Replacing an entire piece of data with a completely new version.
DELETE Throwing a paper in the trash can. Deleting data permanently.
PATCH Putting a band-aid on a small cut. Updating just one tiny piece of data (like changing a user's phone number, but keeping their name).

There are also a few rare methods (OPTIONS, HEAD, TRACE) that servers use to talk to each other behind the scenes to verify connections. We almost never write code for these manually.

Use this when: We are designing our API. We should match the appropriate HTTP method to the action the user wants to perform.

Don't use this when: We are trying to use a GET request to create or delete data. This breaks how the internet fundamentally works!

The Decorator Syntax Explained

Now that we understand the theory, let's look at how we actually wire up a path operation in Python.

from fastapi import FastAPI

app = FastAPI()

@app.post("/login")
def login_user():
    return {"message": "We are logged in!"}

This looks simple, but there is a lot of hidden power here. Let's break down exactly what the decorator does.

Line 5: The @app.post("/login") is the decorator. Let's split it into three pieces:

  • @app: The @ symbol tells Python to wrap the function underneath it with some extra logic. app is the name of our FastAPI instance.
  • .post: This is the HTTP method we chose from our table above. We are creating data (logging in), so we use POST.
  • ("/login"): This is the exact path or URL the user must visit.

Line 6: The Python function directly below the decorator is the code that will actually run when a matching request arrives.

⚠️ Common Mistake: Forgetting the leading slash. Writing "login" instead of "/login" will cause errors. Every path must start with a forward slash!

Order Matters!

FastAPI reads our file from top to bottom. As soon as it finds a path that perfectly matches the user's request, it stops looking and runs the attached function.

Because of this, we must always put specific, hardcoded paths before dynamic, flexible paths. 

@app.get("/users/me")
def read_current_user():
    return {"user_id": "the current user"}

@app.get("/users/{user_id}")
def read_user(user_id: str):
    return {"user_id": user_id}

In this example, if a user visits /users/me, FastAPI checks the first function. It perfectly matches! So it returns the current user. If we swapped the order, the dynamic {user_id} path would catch the request first, and the API would assume "me" is just a random user ID.

In plain English: Always put our specific, hardcoded paths at the top of our file, and our dynamic paths (the ones with curly braces) underneath them.

Configuring Our Operations

Finally, let's explore how to make our automatic documentation look professional. We can configure our path operations by simply passing extra information into the decorator.

The Swagger UI automatically reads this configuration and uses it to organize our manual.

@app.get(
    "/items/",
    tags=["Store Items"],
    summary="Get all items in the store",
    description="This will return a list of every item currently in inventory. It can take a long time to run.",
    deprecated=True
)
def read_items():
    return [{"item": "Apple"}, {"item": "Banana"}]

Let's look at what each of these configuration options does:

tags: This is a list of strings. FastAPI groups all path operations with the same tag together visually in the documentation. This is amazing for keeping large APIs organized.

summary: A short, one-sentence description of what this route does. It appears right next to the path in the Swagger UI.

description: A longer, detailed explanation. This shows up when the user clicks the route to expand it in the documentation. It fully supports Markdown styling.

deprecated=True: If we are planning to delete a route in the future, we can mark it as deprecated. It will still work, but it will be greyed out in the documentation to warn users to stop using it.
