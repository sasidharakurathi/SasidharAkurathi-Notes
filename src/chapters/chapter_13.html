CHAPTER 13 - Form Data

HTML Forms vs. JSON Bodies

Until now, whenever we wanted to send a large chunk of data to the server (like a user profile), we sent it as a JSON payload. JSON is fantastic when our frontend is a modern React or Vue application.

But what if our frontend is a simple, old-school HTML webpage using standard <form> tags? When a user clicks "Submit" on a standard HTML form, the browser does not create a neat JSON object. Instead, it packages the data using a completely different, much older format called application/x-www-form-urlencoded.

If we try to read an HTML form using the Pydantic JSON skills we learned in Chapter 6, FastAPI will crash and reject the request! We must explicitly tell FastAPI: "Stop looking for JSON. Look for Form data instead."

In plain English: JSON and Form Data are two completely different languages. If our frontend speaks "Form", our backend must explicitly use Form() to understand it.

Receiving Form Data

To tell FastAPI to expect old-school form fields, we import the Form tool from the main fastapi package.

Note: Before using forms in FastAPI, we must quickly run pip install python-multipart in our terminal, as FastAPI relies on this tiny extra package to parse the strange form language.

from fastapi import FastAPI, Form

app = FastAPI()

@app.post("/login/")
def login(username: str = Form(), password: str = Form()):
    return {"username": username}

Line 6: By writing = Form(), we are telling FastAPI that username and password will absolutely not be found in the URL, nor in a JSON body. They will be sent directly by an HTML form.

Form Models with Pydantic

Just like with query parameters and headers, typing out fifty individual form fields would be miserable. We can group our form data into a neat Pydantic model!

from pydantic import BaseModel
from fastapi import FastAPI, Form

app = FastAPI()

class LoginForm(BaseModel):
    username: str
    password: str

@app.post("/login/")
def login(form_data: LoginForm = Form()):
    return form_data

Line 11: We created our Pydantic model normally, but in the route decorator, we use = Form(). FastAPI will magically collect all the scattered HTML form fields and assemble them into a perfectly typed LoginForm object for us.

Use this when: We are building a traditional website where users submit standard HTML forms.

Don't use this when: We are building a modern API designed to communicate with a mobile app or a React frontend. Stick to JSON bodies for modern APIs!

The "Either/Or" Rule (Forms vs JSON)

There is one massive trap when working with forms. We cannot mix an HTML Form and a JSON Body in the exact same request.

It is physically impossible for the browser to send both formats at the same time. The request is either labeled as application/json OR it is labeled as application/x-www-form-urlencoded. It cannot be both.

If we write a route like this, FastAPI will scream at us and the code will fail:

class UserDetails(BaseModel):
    age: int

@app.post("/update/")
def update(
    name: str = Form(),       # This expects Form format
    details: UserDetails      # This expects JSON format
):
    ...

If we truly need to send both complex object data and simple form fields at the exact same time, we have to surrender and send absolutely everything as a Form.

There is one major exception to this rule: File Uploads! File uploads use a special type of "Multipart Form", which we will explore in the very next chapter.
