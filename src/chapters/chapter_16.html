What is Dependency Injection?

Dependency Injection (often just called DI) sounds like a terrifyingly complex computer science concept. But in reality, it is incredibly simple. We use it in real life every single day.

Imagine we walk into a coffee shop and order a latte. Does the barista ask us to go to the back room, roast the beans, boil the water, pull the espresso shot, and steam the milk? Of course not. We simply ask for a "latte", and the barista injects (hands) the finished latte to us.

In our code, our route functions are the angry customers. They do not want to connect to the database, parse security tokens, or figure out pagination logic themselves. They just want the final result. Dependency Injection allows our routes to declare exactly what they need (like a database connection), and FastAPI magically builds it and hands it to them just before the function runs.

In plain English: Dependency Injection simply means telling FastAPI: "Hey, before you run my route code, go run this other function first, and give me whatever it produces."

Why FastAPI's System is Powerful

Other frameworks have plugins or middleware that try to accomplish this, but they are often difficult to track. With FastAPI, dependencies are just standard Python functions. That means our code editors provide flawless auto-complete, testing them is trivial, and they integrate flawlessly with Swagger documentation.

Our First Dependency

Let's build a very simple dependency. Imagine we have three different routes that all need to know pagination details (how many items to skip, and how many to limit).

from fastapi import Depends, FastAPI

app = FastAPI()

def common_parameters(q: str = None, skip: int = 0, limit: int = 100):
    return {"q": q, "skip": skip, "limit": limit}

@app.get("/items/")
def read_items(commons: dict = Depends(common_parameters)):
    return commons

@app.get("/users/")
def read_users(commons: dict = Depends(common_parameters)):
    return commons

Let's look at how beautifully simple this is!

Lines 5-6: We create a totally normal Python function called common_parameters. It is not a route. It just takes some standard query parameters and returns a dictionary.

How Depends() Actually Works

Let's focus on what happens in the router:

@app.get("/items/")
def read_items(commons: dict = Depends(common_parameters)):
    return commons

Line 2: When a user hits /items/, FastAPI freezes our read_items function. It sees Depends(common_parameters). It immediately takes the incoming HTTP request and passes it to common_parameters.

FastAPI waits for common_parameters to finish extracting the q, skip, and limit variables. Once it returns that dictionary, FastAPI unfreezes our read_items function and injects that exact dictionary into our commons variable!

Shared Logic Across Multiple Routes

Look back at our first example. Notice that both read_items and read_users use the exact same dependency. This is where dependencies shine!

If we ever decide to change our pagination logic (maybe limiting queries to a maximum of 50 items instead of 100), we only have to change the code in one single place (inside common_parameters). All of our routes instantly inherit the new logic without us touching them.

Even better, FastAPI looks inside common_parameters and automatically adds those expected query parameters (q, skip, limit) to both the /items/ and /users/ interactive Swagger documentation independently!

Type Annotations with Dependencies

If we look closely at our example, our type annotation was a bit messy: commons: dict = Depends(...). While this works, we lose autocomplete. Our editor just knows commons is a dictionary, but it doesn't know it holds q, skip, and limit.

We can solve this massively in modern Python 3.9+ by using Annotated!

from typing import Annotated

@app.get("/items/")
def read_items(commons: Annotated[dict, Depends(common_parameters)]):
    return commons

This does the exact same process, but the code reads beautifully. We are saying: "The variable is a dict, and by the way, get it by depending on the common_parameters function." Note that since the dependency function returns a dictionary, our editor still only knows it as a dictionary. Next chapter, we will see how to make dependencies return classes for ultimate autocomplete power!
