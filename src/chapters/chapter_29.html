async def vs def: The Real Difference

In almost every framework before FastAPI (like early Django and Flask), if a user requested data from the database, the entire server stopped to wait for the database to reply. If 100 users asked at once, the server serviced them one by one, like a single incredibly stressed cashier at a grocery store.

FastAPI defaults to Asynchronous Concurrency. This is like a single cashier who is incredibly smart: while they are waiting for User A's credit card to process, they don't just stand there staring at the screen. They instantly pivot to scan User B's items, and then maybe bag User C's items. When User A's receipt finally prints, they pivot back to hand it to them.

By writing async def, we are telling FastAPI: "This function contains code that can be paused (using the await keyword). Feel free to go help other users while I'm stuck waiting."

I/O Bound vs CPU Bound

To master FastAPI, we must memorize the difference between two types of work.

I/O Bound Work (Input/Output)

This is work that requires our computer to wait for an external system. Connecting to a database, making an HTTP request to another API, reading a file from the hard drive, and sending an email are all I/O Bound.

Rule: Always use async def for I/O Bound work.

CPU Bound Work

This is work that requires our computer's processor to sweat. Resizing a 4GB video frame-by-frame, training an AI machine learning model, or calculating a million prime numbers are all CPU Bound.

Rule: Always use normal def for CPU Bound work.

The Thread Pool Magic

If async def is so fast, why would we ever use normal def?

Remember the smart cashier handling 3 customers at once? If one customer drops a massive bag of 1,000 pennies and demands the cashier count them individually (CPU Bound work), the cashier is completely paralyzed. No other customers can be served until all 1,000 pennies are counted.

This is called Blocking the Event Loop. It is the deadliest mistake in FastAPI. If we put heavy CPU work inside an async def, the entire server crashes to a halt for everyone else.

But FastAPI is brilliant. If we write a normal def instead of async def, FastAPI detects it and automatically pushes that function out to an external Thread Pool. It basically hires a temporary, separate cashier just to count those pennies, while the main async cashier continues serving the rest of the line seamlessly!

In plain English: Write async def when waiting. Write def when calculating. FastAPI handles the rest.

Parallel execution with asyncio.gather

What if we need to fetch data from three separate databases to build a user's dashboard? We definitely do not want to wait for them one at a time!

import asyncio
from fastapi import FastAPI

app = FastAPI()

async def get_orders():
    await asyncio.sleep(2) # Fake 2-second DB call
    return ["Guitar", "Keyboard"]

async def get_profile():
    await asyncio.sleep(2) # Fake 2-second DB call
    return {"name": "Alice"}

@app.get("/dashboard")
async def get_dashboard():
    # Run them simultaneously!
    orders, profile = await asyncio.gather(get_orders(), get_profile())
    
    # This route takes 2 seconds total, not 4 seconds!
    return {"orders": orders, "profile": profile}

Line 17: asyncio.gather() fires off both functions at the exact same millisecond. The server finishes the entire route in 2 seconds flat.

Using run_in_executor

Let's say we have an async def route, but deep inside it we are forced to use an old, synchronous library (like an archaic database driver) that does not support await. If we just call the legacy library, it will block the event loop!

We can manually force that specific blocking code into the external thread pool using run_in_executor. 

import time
import asyncio
from fastapi import FastAPI

app = FastAPI()

def archaic_blocking_function():
    time.sleep(3) # A terribly slow, non-async function
    return "Finally done!"

@app.get("/hybrid")
async def smooth_route():
    loop = asyncio.get_running_loop()
    
    # Force the terrible function onto a separate thread
    result = await loop.run_in_executor(None, archaic_blocking_function)
    
    return {"status": result}

Line 16: By passing our old function into the executor, the main FastAPI event loop stays perfectly responsive to other users for the entire 3 seconds.

The "Don't Mix" Rule

As a senior developer, here is the golden rule we must always follow:

⚠️ Never Mix Sync and Async!
If we write an async def route, every single library we call inside it that deals with I/O (like HTTP calls or database queries) MUST natively support the await keyword.

If we stick an old-school requests.get() or a synchronous SQLAlchemy query inside an async def route, our entire server will seize up entirely.

If we absolutely must use a synchronous library (like the standard requests package or Sync SQLAlchemy), we must write our route as a normal def. FastAPI will recognize it and safely put it in the thread pool.

If we want to be blazing fast with async def, we must upgrade our tools (e.g., using httpx instead of requests, and using AsyncSession in SQLAlchemy).
