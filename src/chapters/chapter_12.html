Why response_model Matters

Until now, we have mostly focused on validating the data users send to us. But what about the data we send back? This is arguably even more important!

Imagine a user logs in. Our database returns a giant object containing their username, their email, and their highly secret, encrypted password hash. If we just return that raw object to the browser, we have completely compromised our application's security!

This is where response_model saves the day. It acts like an outbound filter. We tell FastAPI exactly what the final output should look like, and FastAPI will automatically strip away any extra, secret data before it leaves the server.

In plain English: The response_model acts as a secure filter. It guarantees that our API only returns the exact data we promised, hiding any sensitive information we accidentally included from the database.

Using response_model

We do not define the response model inside the function parameters. Instead, we define it right inside the path decorator itself.

from pydantic import BaseModel
from fastapi import FastAPI

app = FastAPI()

class UserIn(BaseModel):
    username: str
    password: str

class UserOut(BaseModel):
    username: str

@app.post("/user/", response_model=UserOut)
def create_user(user: UserIn):
    # We save the user and password to the DB here...
    # Then we return the original object!
    return user

Lines 6-11: We create two distinct models. UserIn requires a password. UserOut does not even have a password field.

Line 13: We attach response_model=UserOut directly to the @app.post decorator.

Line 17: Even though our Python function returns the full user object (which contains the secret password), FastAPI intercepts it on the way out the door, filters it through UserOut, and strips the password completely. It also automatically updates the interactive Swagger documentation to show exactly what this route returns!

response_model_exclude_unset: The Hidden Gem

What if our model has 30 optional fields (like a massive user profile), but the user has only filled out their name and email? Normally, FastAPI will return a massive JSON object with 28 fields set to null.

To save bandwidth and keep our responses clean, we can use a hidden gem parameter.

@app.get("/items/{item_id}", response_model=Item, response_model_exclude_unset=True)
def read_item(item_id: str):
    return items[item_id]

Line 1: By adding response_model_exclude_unset=True, FastAPI incredibly looks at the exact data we sent back. If a field was left blank (unset), FastAPI completely removes it from the final JSON payload rather than sending a messy null.

Include and Exclude

If we want to be incredibly lazy and avoid creating a separate UserOut model, we can simply tell FastAPI to manually exclude or include specific fields on its way out the door.

@app.get("/users/{user_id}", response_model=UserIn, response_model_exclude={"password"})
def read_user(user_id: str):
    return get_user_from_db(user_id)

While this works, creating dedicated models (like UserOut) is highly recommended. It is much easier to maintain, less prone to typos, and results in far better automatic documentation.

Understanding HTTP Status Codes

When our server responds, it does more than just send JSON. It also sends a 3-digit status code. This code instantly tells the browser whether the request succeeded or failed.

Here is the essential cheat sheet for the status codes we will use every day:

  • 200 OK: Everything worked perfectly. (This is FastAPI's default for everything!).
  • 201 Created: Everything worked perfectly, AND we successfully created something new in the database.
  • 204 No Content: Everything worked perfectly, but we have absolutely no data to return (used a lot when deleting things).
  • 400 Bad Request: The user sent us bad data (Pydantic usually triggers this automatically!).
  • 401 Unauthorized: The user is not logged in.
  • 403 Forbidden: The user is not logged in, but also they might be permanently banned or trying to hack us.
  • 404 Not Found: The user asked for a URL or a database item that does not exist.
  • 500 Internal Server Error: Our Python code crashed horribly. The user did nothing wrong; we made a mistake.
⚠️ Common Mistake: Returning error messages inside a 200 OK. Always use real status codes! If a user is not found, do not return {"error": "not found"} with a happy 200 code. Return a real 404 code. Frontend applications rely on these numbers, not text!

The status_code Parameter

As mentioned, FastAPI assumes every successful response should return a 200 OK code. If we are creating a new user or a new task, we should change that to a 201 Created.

We can do this easily inside the route decorator!

from fastapi import FastAPI, status

app = FastAPI()

@app.post("/items/", status_code=status.HTTP_201_CREATED)
def create_item(name: str):
    return {"name": name}

Line 1: We import the status module directly from FastAPI.

Line 5: We use the status_code parameter to manually override the default 200 code with our new 201 code.

Never Hardcode Numbers!

Why did we type out the horribly long status.HTTP_201_CREATED instead of just typing the number 201? Two reasons.

First, memory. Memorizing three-digit numbers is hard. Knowing that 201 means "Created" and 204 means "No Content" takes practice. By typing status.HTTP_, our code editor will instantly pop up a helpful menu with all the named status codes, preventing us from making silly mistakes.

Second, readability. When another developer reads our code six months from now, reading the word "CREATED" is instantly understood. Reading the random number 201 forces them to go look it up.

In plain English: Always use the English labels provided by fastapi.status instead of typing raw status code numbers.

Additional Status Codes in Documentation

If our route normally returns a 200 OK, but might occasionally return a 404 Not Found (if the ID doesn't exist), we can officially document that possibility so frontend developers know what alternate responses to expect!

@app.get(
    "/items/{item_id}", 
    responses={404: {"description": "The item was not found"}}
)
def read_item(item_id: str):
    ...

Line 3: We use the responses dictionary inside the decorator to define exactly what extra status codes this route is capable of throwing. This will cleanly appear in our Swagger UI manual!
