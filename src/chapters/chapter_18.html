Setting Up OAuth2PasswordBearer

Now that we understand the concept of a "valet key" (a token), how do we actually tell FastAPI to demand one? We use a brilliant built-in tool called OAuth2PasswordBearer.

This tool does one massive job for us: it searches the incoming HTTP request specifically for an Authorization header, and looks for a token that starts with the word "Bearer". 

from fastapi import Depends, FastAPI
from fastapi.security import OAuth2PasswordBearer

app = FastAPI()

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")

@app.get("/items/")
def read_items(token: str = Depends(oauth2_scheme)):
    return {"token": token}

Line 6: We create our security scheme. Notice we passed tokenUrl="token". This does NOT create the login route for us. It simply tells FastAPI's Swagger documentation: "Hey, when a user clicks the Authorize button, send their username and password to the /token path to get the key."

Line 9: We protect our route by adding = Depends(oauth2_scheme). Now, if a user tries to visit /items/ without a valid Bearer token, FastAPI will automatically block them and return a beautiful 401 Unauthorized JSON error. If they do provide the token, FastAPI hands it to our function as a simple string!

In plain English: OAuth2PasswordBearer ensures nobody gets into our route without presenting a token.

The Fake User Database

To demonstrate how this works end-to-end, we need a fake database. We will use a simple Python dictionary where the usernames are the keys, and the values are their full profiles (including an email and a name).

fake_users_db = {
    "johndoe": {
        "username": "johndoe",
        "full_name": "John Doe",
        "email": "johndoe@example.com",
    },
    "alice": {
        "username": "alice",
        "full_name": "Alice Wonderland",

}

The get_current_user Master Pattern

Right now, our oauth2_scheme only returns a raw token string (like "12345"). That is not very helpful. We want to know exactly which user owns that token!

This introduces the most important dependency pattern in FastAPI: get_current_user.

We write a special dependency function that takes the raw token, looks it up in our database, and returns the actual User dictionary (or Pydantic model).

from pydantic import BaseModel
from fastapi import HTTPException, status

class User(BaseModel):
    username: str
    email: str | None = None
    full_name: str | None = None

def decode_token(token: str):
    # Imagine this function decodes a Real JWT. For now, 
    # we pretend the user just sent their username as the token.
    return token

async def get_current_user(token: str = Depends(oauth2_scheme)):
    # 1. Decode the token to discover who they are
    username = decode_token(token)
    
    # 2. Look them up in our database
    user_dict = fake_users_db.get(username)
    
    # 3. If they don't exist, instantly ban them
    if not user_dict:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid authentication credentials",
        )
        
    # 4. Return the perfect Pydantic model
    return User(**user_dict)

Protecting a Route

Now that we have built our magnificent get_current_user dependency, protecting any route in our entire application becomes incredibly easy and clean.

from typing import Annotated

@app.get("/users/me")
async def read_users_me(
    current_user: Annotated[User, Depends(get_current_user)]
):
    return current_user

Let's admire this masterpiece:

  • We tell the route to depend on get_current_user.
  • FastAPI automatically realizes that get_current_user depends on oauth2_scheme.
  • FastAPI checks the incoming HTTP headers.
  • It extracts the Bearer token.
  • It passes the token to get_current_user.
  • get_current_user decodes it, looks up "johndoe" in the database, and creates a User Pydantic model.
  • FastAPI finally unfreezes our route function and hands us the fully secure, fully verified current_user object!

If any of those steps fail (missing token, bad header, user deleted from database), FastAPI instantly kicks the user out with a 401 error. Our highly secure route never even runs unless the user is completely verified!

Use this when: We need to secure an endpoint. By simply adding Depends(get_current_user) to any route, we instantly lock it down and simultaneously gain access to all the user's data. It is the ultimate two-for-one tool.
