Why Good Examples Matter

Imagine opening a complex form and seeing a dozen blank text boxes. Without any hints, we have to guess what data the API expects. Does it want a full name or just a first name? A numeric ID or a string? A good API prevents this confusion.

By adding concrete examples to our documentation, we make our API 10x easier to use. When other developers open our Swagger UI (/docs), the text boxes will pre-fill with realistic, working data. They can just click the "Execute" button and run a successful test immediately.

In plain English: We use examples to auto-fill the forms in our interactive documentation. This shows developers exactly what a successful request looks like.

Adding Examples to Pydantic Models

The most robust way to add an example is by attaching it directly to our Pydantic model. When we do this, the example automatically appears everywhere that model is used across our entire API.

from pydantic import BaseModel

class Item(BaseModel):
    name: str
    description: str = None
    price: float
    tax: float = None

    model_config = {
        "json_schema_extra": {
            "examples": [
                {
                    "name": "Foo",
                    "description": "A very nice Item",
                    "price": 35.4,
                    "tax": 3.2,
                }
            ]
        }
    }

Let's look at how this works:

Lines 9-20: We create a special variable inside our class called model_config. Inside that dictionary, we use the json_schema_extra key to inject our examples.

Now, whenever someone looks at the documentation for a route that expects an Item, the entire JSON body will be neatly pre-filled with the exact "Foo" item we defined above.

⚠️ Note for older code: If we look at older FastAPI tutorials, we might see them using an internal class Config: block with a schema_extra key. The model_config dictionary is the modern, updated way to do this in the newest versions of FastAPI.

Using Examples in Field(), Path(), and Query()

If we do not want to add examples to the entire model at once, we can get granular. We can attach examples directly to individual fields using the `Field()` function.

from pydantic import BaseModel, Field

class Item(BaseModel):
    name: str = Field(examples=["Foo"])
    description: str = Field(default=None, examples=["A very nice Item"])
    price: float = Field(examples=[35.4])
    tax: float = Field(default=None, examples=[3.2])

This does the exact same thing as the model_config method, but we define the examples piece by piece. It is incredibly clean and easy to read.

We can use this exact same pattern with Path(), Query(), and Body() inside our route definitions.

from fastapi import FastAPI, Body, Path

app = FastAPI()

@app.put("/items/{item_id}")
def update_item(
    item_id: int = Path(examples=[5]),
    item: Item = Body(
        examples=[
            {
                "name": "Foo",
                "price": 35.4,
            }
        ],
    ),
):
    return {"item_id": item_id, "item": item}

Line 7: We tell the documentation that a good example for the path parameter item_id is the number 5.

Lines 8-15: We use Body() to define a full JSON body example directly inside our route function. This example will override any examples defined inside the Pydantic model itself!

Use this when: We want to provide different examples for the same model depending on which route is being used.

openapi_examples: Multiple Named Examples

Giving one example is great. But what if our API behaves differently depending on the input? We might want to show our users an example of a "normal" request, and another example of a "special" request.

We can do this using the highly advanced openapi_examples parameter. This lets us create a drop-down menu in our Swagger documentation where users can switch between multiple named examples.

@app.put("/items/{item_id}")
def update_item(
    item_id: int,
    item: Item = Body(
        openapi_examples={
            "normal": {
                "summary": "A normal example",
                "description": "A **normal** item works correctly.",
                "value": {
                    "name": "Foo",
                    "price": 35.4,
                },
            },
            "invalid": {
                "summary": "Invalid data",
                "description": "This will trigger a validation error.",
                "value": {
                    "name": "Invalid Item",
                    "price": "thirty five",
                },
            },
        },
    ),
):
    return {"item_id": item_id, "item": item}

Let's look at this incredibly powerful tool:

Line 5: We open the openapi_examples dictionary.

Line 6 & 14: We create two distinct, named options: "normal" and "invalid".

Inside each option: We can provide a summary and description (which supports Markdown formatting!) to explain what the example does. We then provide the actual payload inside the value key.

When a developer opens our /docs page for this route, they will see a beautiful dropdown menu labeled "Examples". They can pick "normal" to see a working request, or pick "invalid" to see exactly how our API handles bad data.

In plain English: By spending a few extra minutes adding examples, we dramatically reduce the amount of time we have to spend explaining our API to other developers.
